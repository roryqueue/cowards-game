## Out of Scope

Explicitly excluded from v1.10. Documented to prevent scope creep.

| Feature | Reason |
| --- | --- |
| Broad backend rewrite | v1.10 is a service-boundary consolidation and one-route read-model proof, not a backend ownership swap. |
| Go mutation endpoints, auth/session mutation, ladder entry/schedule/status writes, Match orchestration, job claiming, migrations, persistence writes, or Strategy execution | These require write semantics, auth/session ownership, transaction rules, job ownership, and rollback plans that are not proven by read-model parity. |
| More than one Go read-model route | The milestone decision is exactly one public Strategy route so parity, topology, privacy, and rollback can stay sharp. |
| Production web routing to Go without explicit rollback | Go remains a read-model proof unless required-boundary checks and rollback behavior are explicit. |
| Production hostile-code sandbox promotion | v1.10 must preserve runtime isolation as evidence-only unless a future milestone proves promotion criteria. |
| Counted Python or other non-JS MatchSets, ladders, gauntlets, analytics comparisons, or standings | Non-JS sandbox, package, compatibility, fairness, and product support remain unproven. |
| Public language picker or package dependency ecosystem | These would imply non-JS support parity before promotion criteria are met. |
| Workshop Strategy source, source save, validation/test execution, submission, analytics rerun, or export mutation migration | The v1.10 Workshop slice is read-only analytics/evidence data only. |
| Strategy Revision source retrieval migration | Owner source access is private and source-bearing; v1.10 keeps it out of the read-list migration. |
| Full replay projection or owner-debug replay migration | Replay privacy and Chronicle projection remain too sensitive for this service-boundary slice. |
| Node `vm` or `vm2` as a security boundary | Node `vm` is not a security mechanism for hostile Strategy code. |
| Rule, Chronicle, scoring, terminology, engine, or deterministic runtime semantics changes | Boundary work must preserve deterministic gameplay and existing replay/evidence compatibility. |
| Durable ratings, official public tournaments, custom arenas, monetization, or live model/human control | These do not serve the v1.10 boundary-completion goal. |
