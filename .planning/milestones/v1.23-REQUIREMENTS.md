# Requirements: Coward's Game v1.23

**Defined:** 2026-05-25
**Core Value:** Players can design, run, replay, and understand deterministic autonomous doctrines competing under the canonical Coward's Game rules.

## Milestone Goal

Attempt to promote Rust and Zig from non-counted exhibition alpha to non-counted exhibition beta, gated by real signed-in multi-compiler proof, runtime hardening evidence, Zig ergonomics, ABI evidence, privacy, and conservative split-capable promotion decisions.

## Baseline

- v1.22 is complete, committed, tagged `v1.22`, and archived.
- Active `.planning/REQUIREMENTS.md` was absent before this milestone began.
- JS/TS remains the counted Strategy path.
- Python remains non-counted exhibition beta.
- Rust and Zig begin v1.23 as non-counted exhibition alpha only.
- WASM/WASI hardening evidence passes 19/19 probes in v1.22.
- Zig compiles no-std `wasm32-wasi` source to immutable WASM artifact metadata and executes through Wasmtime in v1.22 proof.
- Rust compiles to immutable WASM/WASI artifact metadata and executes through Wasmtime.
- Preview 1 stdin/stdout JSON remains the only active execution ABI unless a future proof explicitly promotes another ABI.
- Full signed-in JS/TS/Rust/Zig live MatchSet proof was not rerun in v1.22 and is a v1.23 gate.

## v1 Requirements

### Baseline and Beta Criteria

- [x] **BASE-01**: Operator can see v1.22 evidence summarized as the v1.23 regression floor before implementation changes begin.
- [x] **BASE-02**: Operator can see explicit non-counted exhibition beta criteria for Rust and Zig, including signed-in proof, artifact execution, hardening, privacy, replay, no-fallback, and JS/TS regression gates.
- [x] **BASE-03**: Operator can see explicit split promotion outcomes: Rust beta / Zig alpha, both beta, neither beta, or both remain alpha.
- [x] **BASE-04**: Product and planning docs make clear that beta means non-counted exhibition beta only.
- [x] **BASE-05**: JS/TS counted Strategy support remains intact and has a regression gate before any Rust/Zig promotion decision.
- [x] **BASE-06**: Python remains non-counted exhibition beta and is not reclassified by Rust/Zig promotion work.

### Zig Ergonomics

- [x] **ERG-01**: User can author or load a safer Zig starter/helper layer that is more ergonomic than the v1.22 tiny no-std proof.
- [x] **ERG-02**: Operator can inspect Zig helper import/capability evidence before any helper is exposed as beta-ready.
- [x] **ERG-03**: Zig helper validation fails loud when compiled output imports forbidden filesystem, network, clock/random, path, poll, environment, package, or host capability surfaces.
- [x] **ERG-04**: Workshop Zig samples remain honest about no-std or helper limitations and do not imply counted, ranked, ladder, or production sandbox support.
- [x] **ERG-05**: Zig beta eligibility can remain alpha independently from Rust if ergonomics or capability evidence is insufficient.

### Rust/Zig Runtime Hardening

- [x] **HARD-01**: Runtime-service can rerun Rust WASM/WASI compile, artifact hash, and Wasmtime execution evidence against the v1.23 baseline.
- [x] **HARD-02**: Runtime-service can rerun Zig WASM/WASI compile, artifact hash, and Wasmtime execution evidence against the v1.23 baseline.
- [x] **HARD-03**: Operator can inspect Rust/Zig hardening probes for timeout/fuel, memory caps, stdio/result caps, malformed JSON, invalid schema, traps, stale artifacts, and missing artifacts.
- [x] **HARD-04**: Runtime failures distinguish Strategy failure from system failure and remain public-safe by default.
- [x] **HARD-05**: Runtime-service refuses stale hash, missing artifact, or runtime mismatch without source execution fallback.
- [x] **HARD-06**: Runtime hardening evidence remains candidate/beta readiness evidence and does not claim production sandbox certification.

### ABI Readiness

- [x] **ABI-01**: Operator can inspect a current Preview 1 stdin/stdout JSON ABI evidence record that confirms the active execution path remains schema-validated and capped.
- [x] **ABI-02**: Operator can inspect a direct-export proof spike or fail-loud decision covering memory ownership, allocation, string/buffer passing, caps, schema validation, and Rust/Zig parity.
- [x] **ABI-03**: Operator can inspect a component model/WIT proof spike or fail-loud decision covering WIT world shape, language support, Wasmtime execution, caps, schema validation, and Rust/Zig parity.
- [x] **ABI-04**: No ABI proof spike silently becomes a Match execution path without an explicit promotion decision.
- [x] **ABI-05**: Public and planning evidence explain why Preview 1 JSON stays active or what would be required to migrate later.
- [x] **ABI-06**: ABI evidence includes rollback and compatibility notes for artifact metadata produced before and after the spike.

### Signed-In Multi-Compiler Proof

- [x] **PROOF-01**: Signed-in user can save one JS/TS Strategy Revision.
- [x] **PROOF-02**: Signed-in user can save one Rust WASM/WASI Strategy Revision backed by immutable artifact metadata.
- [x] **PROOF-03**: Signed-in user can save one Zig WASM/WASI Strategy Revision backed by immutable artifact metadata.
- [x] **PROOF-04**: Signed-in user can run a JS/TS-vs-Rust non-counted exhibition MatchSet.
- [x] **PROOF-05**: Signed-in user can run a Rust-vs-Rust non-counted exhibition MatchSet.
- [x] **PROOF-06**: Signed-in user can run a Rust-vs-Zig non-counted exhibition MatchSet.
- [x] **PROOF-07**: Signed-in user can run a Zig-vs-Zig non-counted exhibition MatchSet.

### UX Labels and Privacy

- [x] **UX-01**: Workshop language selectors, validation output, save paths, and samples show Rust/Zig as alpha or non-counted exhibition beta according to actual evidence.
- [x] **UX-02**: Exhibition creation flow prevents Rust/Zig from entering counted, ranked, ladder, or gauntlet paths.
- [x] **UX-03**: MatchSet result pages show public-safe language/runtime/artifact evidence and non-counted status for Rust/Zig exhibitions.
- [x] **UX-04**: Replay pages show plausible full Match starts with visible Soldiers and terrain inside declared board bounds.
- [x] **UX-05**: Public result/replay output does not expose Strategy source, StrategyMemory, SoldierMemory, objective payloads, owner debug, raw diagnostics, host paths, env values, tokens, DB DSNs, package paths, or private runtime internals by default.

### Boundary Monitors and Compatibility

- [x] **MON-01**: Boundary monitors fail when Rust, Zig, Python, or TypeScript attempt backend ownership creep.
- [x] **MON-02**: Boundary monitors fail when Strategy code can execute in web/API/Go.
- [x] **MON-03**: No-fallback drills prove stopped, unavailable, stale, or mismatched Rust/Zig runtime paths fail loud rather than falling back silently.
- [x] **MON-04**: Artifact compatibility evidence records target triple, WASI profile, ABI envelope, toolchain version, hash, byte count, validation status, and retention/rollback behavior.
- [x] **MON-05**: Public evidence and monitor artifacts are safe to publish without private Strategy data or host/runtime secrets.

### Promotion Decision and Closure

- [x] **DEC-01**: Operator can inspect a final v1.23 promotion decision that chooses one of the allowed split outcomes.
- [x] **DEC-02**: Promotion decision explicitly preserves JS/TS counted support, Python non-counted exhibition beta, and no counted/ranked/ladder/gauntlet Rust/Zig/WASM support.
- [x] **DEC-03**: Milestone audit verifies requirements coverage, code review findings, UI/privacy review findings, signed-in proof, replay plausibility, and promotion evidence before closure.
- [x] **DEC-04**: v1.23 planning artifacts are archived, active requirements are removed at milestone close, and commit/tag evidence is recorded.

## Future Requirements

### Runtime and ABI

- **FUT-ABI-01**: Strategy execution can migrate to direct exports if memory ownership, per-language glue, validation, caps, privacy, and rollback evidence pass in a dedicated promotion milestone.
- **FUT-ABI-02**: Strategy execution can migrate to component model/WIT if Rust/Zig toolchains, WIT worlds, Wasmtime hosting, validation, compatibility, and rollback evidence pass in a dedicated promotion milestone.
- **FUT-SBX-01**: WASM/WASI can be considered for production sandbox certification only after a dedicated hostile-code, operational, abuse, isolation, and deployment proof.
- **FUT-LANG-01**: Rust/Zig can be considered for counted play only after beta evidence, governance, deterministic compatibility, sandbox, privacy, ranking, replay, rollback, and abuse criteria are proven.

## Out of Scope

| Feature | Reason |
| --- | --- |
| Rust/Zig counted, ranked, ladder, gauntlet, or durable rating eligibility | v1.23 targets only non-counted exhibition beta evidence. |
| WASM/WASI production sandbox certification | v1.23 can improve readiness evidence but cannot certify production hostile-code isolation. |
| Direct-export or component-model production execution path | v1.23 may spike these ABIs but must not silently replace Preview 1 JSON. |
| Rust, Zig, Python, or TypeScript backend ownership | Go owns orchestration, persistence-facing backend behavior, Match lifecycle, scoring, public evidence, retry policy, and promotion decisions. |
| Strategy execution in web/API/Go | Hostile Strategy code must stay behind runtime-service / Runtime Broker boundaries. |
| Arbitrary Cargo, Zig package, or third-party dependency support | Package policy and supply-chain risk require separate design and proof. |
| Broad production multi-language marketplace | v1.23 is a narrow exhibition beta readiness milestone, not a marketplace launch. |
