# Requirements: Coward's Game v1.25

**Defined:** 2026-05-30
**Core Value:** Players can design, run, replay, and understand deterministic autonomous doctrines competing under the canonical Coward's Game rules.

## Milestone Goal

Establish a firm, versioned, tested boundary between Match execution and the app around it, so execution internals and app/result/replay UX can iterate in parallel with confidence.

## Baseline

- v1.24 is complete, committed, tagged `v1.24`, and archived.
- Active `.planning/REQUIREMENTS.md` was absent before this milestone began.
- This is a core contract milestone, not a runtime promotion milestone.
- JS/TS remains the counted Strategy path.
- Python, Rust, and Zig remain non-counted exhibition beta.
- No new language promotion, production sandbox certification, direct-export ABI migration, Component Model/WIT execution ABI migration, counted non-JS play, or broad ranking/ladder expansion is allowed in this milestone.
- WASI Preview 1 stdin/stdout JSON remains the active WASM/WASI execution ABI unless a future explicit decision promotes a replacement.
- Go owns orchestration, persistence-facing backend behavior, Match lifecycle, scoring, public evidence, retry policy, and promotion decisions.
- Runtime-service owns hostile Strategy execution only through schema-validated ABI envelopes and registered runtime implementations.
- Strategy code must not execute in web/API/Go.
- Public output must not expose Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw diagnostics, host paths, env values, tokens, DB details, package paths, or private runtime internals by default.

## v1 Requirements

### Baseline and Inventory

- [x] **BASE-01**: Operator can inspect a v1.24 baseline summary before implementation begins.
- [x] **BASE-02**: Operator can inspect an inventory of current MatchSet creation, Match execution, result, replay, runtime evidence, failure evidence, and public/private evidence DTOs.
- [x] **BASE-03**: Operator can inspect an inventory of app pages, API routes, service adapters, Go handlers, runtime-service envelopes, fixtures, and tests that depend on Match execution internals.
- [x] **BASE-04**: Inventory classifies each surface as app-facing contract, owner/test-only contract, execution-internal, persistence-internal, or intentionally unstable.
- [x] **BASE-05**: Planning and inventory artifacts explicitly preserve JS/TS counted support, Python/Rust/Zig non-counted exhibition beta status, Preview 1 JSON ABI status, Go ownership, runtime-service hostile execution ownership, and public-output privacy.

### Lifecycle Contract

- [x] **LIFE-01**: Operator can inspect a canonical Match lifecycle state machine covering queued, accepted, running, complete, failed, degraded, unavailable, retryable, and non-retryable semantics.
- [x] **LIFE-02**: Operator can inspect a canonical MatchSet lifecycle state machine that composes Match states into public result states without leaking execution internals.
- [x] **LIFE-03**: Lifecycle contract distinguishes strategy failure, system failure, timeout, unavailable runtime, malformed runtime result, stale artifact, blocked, missing Chronicle, and public-safe no-result states.
- [x] **LIFE-04**: Lifecycle contract defines which states are terminal, which are retryable, which are degraded terminal public evidence, and which remain pending/running.
- [x] **LIFE-05**: Go orchestration tests or contract tests prove backend lifecycle outputs conform to the canonical state vocabulary.
- [x] **LIFE-06**: App evidence copy and result/replay pages consume lifecycle semantics through the contract instead of inferring from runtime internals.

### App-Facing DTO Schemas

- [x] **DTO-01**: Versioned app-facing DTO schemas exist for MatchSet summary/result and are distinct from execution-internal runtime-service responses.
- [x] **DTO-02**: Versioned app-facing DTO schemas exist for replay metadata and replay evidence.
- [x] **DTO-03**: Versioned app-facing DTO schemas exist for runtime evidence and failure evidence categories that are public-safe by default.
- [x] **DTO-04**: DTO schemas explicitly model public/private evidence splits, including owner/test-only fields that must never appear in default public DTOs.
- [x] **DTO-05**: DTO schemas preserve runtime labels and eligibility semantics without exposing Strategy source, memory, objective payloads, raw diagnostics, host paths, env values, tokens, DB details, package paths, or private runtime internals.
- [x] **DTO-06**: Generated or committed schema artifacts are validated against examples, fixtures, OpenAPI/service metadata where applicable, and Go fixture outputs.
- [x] **DTO-07**: Unknown, stale, malformed, or unversioned DTO payloads fail closed in contract tests or adapters.

### Fixture Catalog and Golden States

- [x] **FIX-01**: Fixture catalog includes a complete MatchSet/result scenario with replay evidence.
- [x] **FIX-02**: Fixture catalog includes running and queued MatchSet/result scenarios.
- [x] **FIX-03**: Fixture catalog includes strategy failure and system failure scenarios with distinct public evidence.
- [x] **FIX-04**: Fixture catalog includes timeout, unavailable runtime, malformed runtime result, and stale artifact scenarios.
- [x] **FIX-05**: Fixture catalog includes a public-safe replay scenario with plausible sequence-0 board state, in-bounds visible Soldiers and terrain, and no private fields.
- [x] **FIX-06**: Fixtures are schema-validated and golden enough for app/result/replay tests to use without live execution services.
- [x] **FIX-07**: Fixture catalog documents which scenarios are public, owner/test-only, execution-internal, or intentionally unstable.

### Contract Tests and Drift Monitors

- [x] **TEST-01**: Contract tests prove Go public MatchSet summary/result outputs match the versioned DTO schemas.
- [x] **TEST-02**: Contract tests prove Go public replay metadata and replay evidence outputs match the versioned DTO schemas.
- [x] **TEST-03**: Contract tests prove runtime-service failure envelopes are translated into app-facing failure evidence without leaking private diagnostics.
- [x] **TEST-04**: App tests render result pages against every fixture scenario without private fields or execution-internal terms.
- [x] **TEST-05**: App tests render replay pages against fixture replay evidence without private fields, clipped boards, off-screen pieces, or blank canvas output.
- [x] **TEST-06**: Drift monitors fail when lifecycle vocabulary, DTO versions, fixture scenario coverage, public/private evidence boundaries, or Go/runtime-service ownership boundaries drift.
- [x] **TEST-07**: Privacy scans cover schema artifacts, fixture JSON, Markdown evidence, Go outputs, and rendered app pages.

### Replay and Result Interface Hardening

- [x] **UI-01**: Result page consumes MatchSet/result DTOs through the frozen app contract or fixture adapter rather than execution internals.
- [x] **UI-02**: Replay page consumes replay metadata/evidence DTOs through the frozen app contract or fixture adapter rather than execution internals.
- [x] **UI-03**: Result and replay evidence copy explains queued, running, complete, degraded, failed, unavailable, retryable, and non-retryable states using public-safe language.
- [x] **UI-04**: Public result/replay pages omit Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw diagnostics, host paths, env values, tokens, DB details, package paths, and private runtime internals by default.
- [x] **UI-05**: Owner/test-only replay debug surfaces remain gated and are not part of the default public contract.
- [x] **UI-06**: Browser or Playwright proof validates representative fixture-backed result/replay pages across public-safe scenarios.

### Parallelization Adapter

- [x] **ADAPT-01**: App/test code can switch to a fixture-backed adapter for MatchSet result and replay flows without live Match execution services.
- [x] **ADAPT-02**: Fixture-backed adapter is gated to test/local development modes and cannot become a silent production fallback.
- [x] **ADAPT-03**: Adapter returns the same versioned DTO shapes as Go/service-backed reads.
- [x] **ADAPT-04**: Adapter supports complete, queued, running, strategy failure, system failure, timeout, unavailable runtime, malformed runtime result, stale artifact, and public-safe replay scenarios.
- [x] **ADAPT-05**: Adapter documentation explains how app/result/replay work can proceed in parallel with execution changes and where live proof remains required.

### End-to-End Proof and Freeze Decision

- [x] **PROOF-01**: Fixture-mode proof renders the result/replay scenario catalog and records public-safe evidence.
- [x] **PROOF-02**: Signed-in live proof still creates eligible JS/TS counted and non-JS non-counted exhibition evidence without Strategy execution in web/API/Go.
- [x] **PROOF-03**: Live proof opens result and replay pages and verifies lifecycle copy, DTO parsing, replay plausibility, and public-output privacy.
- [x] **PROOF-04**: Final decision states whether the app-facing Match execution interface is frozen enough for execution and app work to proceed in parallel.
- [x] **PROOF-05**: Final decision lists intentionally unstable internals, including runtime-service implementation details, execution ABI promotion candidates, private diagnostics, owner/test-only debug payloads, and future sandbox/promotion claims.

### Audit and Closure

- [x] **CLOSE-01**: Code review verifies lifecycle, DTO, fixture, adapter, privacy, ownership, and monitor changes before closure.
- [x] **CLOSE-02**: Validation verifies requirement coverage, contract tests, app fixture rendering, live proof, replay board realism, privacy scans, and freeze decision.
- [x] **CLOSE-03**: Final milestone decision states that v1.25 does not promote runtimes, certify a production sandbox, or migrate execution ABI.
- [x] **CLOSE-04**: v1.25 planning artifacts are archived, active requirements are removed at milestone close, and commit/tag evidence is recorded.

## Future Requirements

### Match Execution and UX Parallel Work

- **FUT-CONTRACT-01**: Future execution internals can change behind the frozen app-facing contract only when DTO, lifecycle, fixture, and monitor compatibility remains intact or a new contract version is explicitly promoted.
- **FUT-CONTRACT-02**: Future app/result/replay UX can add richer views only through public-safe DTO additions or explicitly gated owner/test-only contracts.
- **FUT-LIVE-01**: Future production-sandbox, ABI migration, counted non-JS, ranked, ladder, or gauntlet work must run its own proof and must not rely on v1.25 interface freeze as promotion evidence.

## Out of Scope

| Feature | Reason |
| --- | --- |
| New Strategy language promotion | v1.25 freezes app/execution contracts; language eligibility remains unchanged. |
| Production sandbox certification | v1.25 is not a runtime promotion or deployment-hardening milestone. |
| Direct-export or Component Model/WIT execution ABI migration | Preview 1 stdin/stdout JSON remains active until a future explicit promotion decision. |
| Broad ranking, ladder, gauntlet, or durable rating expansion | This milestone protects Match execution interfaces, not competitive product scope. |
| Runtime-service becoming backend owner | Runtime-service owns hostile execution only; Go owns orchestration, lifecycle, scoring, public evidence, retry policy, and promotion decisions. |
| Go/web/API Strategy execution | Hostile Strategy code must stay behind runtime-service / Runtime Broker boundaries. |
| Public raw diagnostics or private runtime internals | Public evidence must remain source-free and private-data safe. |
| Silent fixture fallback in production | Fixture-backed adapters are for test/local app development only. |
