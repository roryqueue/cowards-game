# Plan 04-03: Worker-Only Runtime Execution Adapter

<objective>
Implement the worker-only executable runtime entrypoint that adapts validated Strategy Revisions to the existing synchronous `StrategyRuntime` interface and maps all sandbox failures to typed runtime violations.
</objective>

<requirements_addressed>
RUN-03, RUN-04, RUN-05, RUN-06, RUN-07, RUN-08, RUN-09, RUN-10, TEST-04
</requirements_addressed>

<threat_model>
| Threat | Severity | Mitigation |
|--------|----------|------------|
| T-04-07: Executable runtime is imported by web or engine code | high | Add `@cowards/runtime-js/worker` as a separate export and block it from `apps/web`, `packages/engine`, and non-worker package rules. |
| T-04-08: Runtime exceptions escape into engine execution | high | Catch thrown exceptions and return `RuntimeViolationType: "THROWN_EXCEPTION"` with owner-only raw details. |
| T-04-09: Invalid or oversized output mutates memory | high | Validate full outputs with `StrategyResultSchema` and `SoldierBrainResultSchema` before returning success. |
| T-04-10: Prototype sandbox is mistaken for production-grade isolation | medium | Document and test guardrails while keeping a replaceable `StrategyRuntime` adapter boundary. |
</threat_model>

<task id="04-03-03" type="execute">
  <title>Implement worker bridge for synchronous engine calls</title>
  <read_first>
    - packages/runtime-js/src/revision.ts
    - packages/runtime-js/src/transpile.ts
    - packages/runtime-js/src/validation.ts

    - packages/engine/src/activation.ts
  </read_first>
  <action>
    Add `packages/runtime-js/src/worker-harness.ts` as the isolated one-shot execution script used by Node `worker_threads`. The harness must receive worker data containing `{ source, methodName, input, port, signalBuffer }`, compile already-validated source without using `node:vm`, call exactly one method, post `{ ok: true, value }` or `{ ok: false, violation }` to the transferred `MessagePort`, set `Atomics.store(signal, 0, 1)`, call `Atomics.notify(signal, 0)`, and then exit.

    The harness must shadow or block these names when compiling/calling strategy code: `eval`, `Function`, `process`, `require`, `fetch`, `WebAssembly`, `Worker`, `Date`, `setTimeout`, `setInterval`, `setImmediate`, and `Math.random`. If a blocked value is accessed or called, return `RuntimeViolationType: "FORBIDDEN_CAPABILITY"`.

    Add `packages/runtime-js/src/worker-bridge.ts` exporting:
    - `runStrategyMethodInWorker(args: { source: string; methodName: "selectActivations" | "soldierBrain"; input: unknown; timeoutMs?: number | undefined }): RuntimeResult<unknown>`

    `runStrategyMethodInWorker` must:
    - create a `SharedArrayBuffer` and `MessageChannel`
    - spawn `new Worker(new URL("./worker-harness.js", import.meta.url), { workerData, transferList: [port2], env: {}, execArgv: [], resourceLimits: { maxOldGenerationSizeMb: 16, maxYoungGenerationSizeMb: 8, stackSizeMb: 4 } })`
    - wait synchronously with `Atomics.wait(signal, 0, 0, timeoutMs ?? RUNTIME_TIMEOUT_MS)` so it can satisfy the existing synchronous engine `StrategyRuntime` interface
    - on timeout, call `worker.terminate()` and return `RuntimeViolationType: "TIMEOUT"`
    - after notification, call `receiveMessageOnPort(port1)` to synchronously retrieve the result
    - map missing/invalid worker messages to `RuntimeViolationType: "THROWN_EXCEPTION"`

    This worker bridge is the Phase 4 replaceable boundary. It must not use Node `vm`, filesystem, network, database, or process environment access for strategy execution.
  </action>

<task id="04-03-04" type="test">
  <title>Implement StrategyRuntime execution adapter and failure matrix tests</title>
  <read_first>
    - packages/runtime-js/src/worker-bridge.ts
    - packages/runtime-js/src/revision.ts
    - packages/spec/src/schemas.ts
    - packages/engine/src/types.ts
  </read_first>
  <action>
    Add `packages/runtime-js/src/executor.ts` exporting:
    - `createRuntimeFromRevision(revision: StrategyRevision): StrategyRuntime`

    `createRuntimeFromRevision` must:
    - require `revision.validation.valid === true` before execution; otherwise both methods return `INVALID_OUTPUT` with message `Strategy Revision is not valid`
    - call `runStrategyMethodInWorker({ source: revision.source, methodName: "selectActivations", input })` for activation selection
    - call `runStrategyMethodInWorker({ source: revision.source, methodName: "soldierBrain", input })` for SoldierBrain
    - parse `selectActivations` results through `StrategyResultSchema.safeParse`
    - parse `soldierBrain` results through `SoldierBrainResultSchema.safeParse`
    - reject returned Promises or non-plain results as `INVALID_OUTPUT`
    - return `violation("INVALID_OUTPUT", ...)` for schema failures
    - preserve worker bridge `TIMEOUT`, `THROWN_EXCEPTION`, and `FORBIDDEN_CAPABILITY` violations
    - return `violation("OVERSIZED_OUTPUT", ...)` when schema failure is caused by StrategyMemory, SoldierMemory, or objective payload byte limits

    The adapter must not update StrategyMemory or SoldierMemory itself. It only returns parsed success values or typed violations to the engine; the engine remains the only owner of memory state updates.

    Add `packages/runtime-js/src/executor.test.ts` with tests proving:
    - valid `selectActivations` returns `{ ok: true }` with activation orders and StrategyMemory
    - valid `soldierBrain` returns `{ ok: true }` with exactly one Action and SoldierMemory
    - invalid `selectActivations` output returns `INVALID_OUTPUT`
    - invalid `soldierBrain` output returns `INVALID_OUTPUT`
    - oversized StrategyMemory returns `OVERSIZED_OUTPUT`
    - oversized SoldierMemory returns `OVERSIZED_OUTPUT`
    - oversized objective payload from activation selection returns `OVERSIZED_OUTPUT`
    - thrown exception returns `THROWN_EXCEPTION`
    - infinite loop or timeout fixture returns `TIMEOUT`
    - forbidden access attempt returns `FORBIDDEN_CAPABILITY`
    - async/Promise return is rejected as `INVALID_OUTPUT`
    - no successful result is returned until the full output schema parse succeeds
  </action>
