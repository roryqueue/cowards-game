# Plan 01-02: Package and App Skeleton Boundaries

<objective>
Create skeletal apps and package shells with dependency direction matching Phase 1 decisions.
</objective>

<requirements>
- FOUND-02: Developer can run a single command that starts the web app, worker, database, queue dependency, and local sandbox dependencies.
- FOUND-04: The repository enforces package boundaries so UI, worker, engine, runtime, replay, map configs, and shared contracts remain separate.
- FOUND-05: The project records engine, runtime, spec, Chronicle, Strategy Revision, and Arena Variant versions wherever Match reproducibility depends on them.
</requirements>

<must_haves>
- Create `packages/spec`, not `packages/shared`.
- `packages/spec` has no internal workspace dependencies.
- Minimal `apps/web` and `apps/worker` exist and boot without game behavior.
- Package shells exist for future engine, runtime, replay, map-configs, and test-utils boundaries.
- No package imports user strategy execution into `apps/web`.
</must_haves>

<truths>
- D-01: Use `packages/spec` as the only shared contract package in Phase 1.
- D-02: Do not create `packages/shared` in Phase 1.
- D-03: `packages/spec` owns canonical TypeScript types, constants, Zod schemas, event/action/status unions, version IDs, and canonical fixtures.
- D-04: `packages/spec` must not depend on any internal workspace package.
- D-05: Other workspace packages should depend on `packages/spec`; dependency direction should not point back from `spec`.
- D-11: Provide two local development commands: lightweight `pnpm dev` and full `pnpm dev:full`.
- D-12: `pnpm dev` supports a fast loop without forcing all infrastructure every time.
- D-13: `pnpm dev:full` starts the planned full local topology.
- D-14: Docker Compose starts real PostgreSQL and Redis in Phase 1.
- D-15: Create minimal runnable `apps/web` and `apps/worker` in Phase 1.
</truths>

<threat_model>
| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-1-03 | Dependency graph allows rule/runtime code to leak into web app. | high | Package dependencies and lint boundaries prohibit `apps/web` from importing runtime internals. |
| T-1-04 | `packages/spec` becomes a dumping ground with reverse dependencies. | medium | `packages/spec/package.json` has only external dependencies and package exports for contracts. |
</threat_model>
