# Requirements: Coward's Game v1.7 Runtime and Backend Boundary Stabilization

**Defined:** 2026-05-22
**Core Value:** Players can design, run, replay, and understand deterministic autonomous doctrines competing under the canonical Coward's Game rules.

## v1.7 Requirements

Requirements for making Coward's Game ready for a future Go backend and multi-language Strategy runtimes without prematurely rewriting either system.

### Service Boundary Contract

- [x] **SVC-01**: Developer can use typed API/service contracts for auth/session, Strategy revisions, MatchSets, replay DTOs, analytics profiles/runs, exports, ladders, public pages, and health.
- [x] **SVC-02**: Next route handlers and server-facing page loaders move toward a typed client/service layer instead of importing persistence workflows directly.
- [x] **SVC-03**: Service DTO schemas preserve public/privacy-safe output by default for replay, MatchSets, analytics, exports, ladders, profiles, public Strategy cards, and public pages.
- [x] **SVC-04**: Service boundary tests prove existing web flows receive equivalent DTOs after service-layer indirection.
- [x] **SVC-05**: Service contracts distinguish browser-safe public DTOs, owner-authorized DTOs, and internal persistence/runtime records.
- [x] **SVC-06**: Service contracts include explicit compatibility/version fields needed by future remote clients and a future Go backend.

### Strategy Runtime ABI

- [x] **ABI-01**: Developer can use language-neutral JSON request and response envelopes for `selectActivations` and `soldierBrain`.
- [x] **ABI-02**: Runtime ABI schemas validate source/package metadata, input JSON, output JSON, memory limits, objective limits, stdout/stderr caps, timeout settings, and deterministic capability restrictions.
- [x] **ABI-03**: Runtime ABI distinguishes Strategy runtime violations from system failures with stable codes and public-safe messages.
- [x] **ABI-04**: Runtime ABI includes version negotiation for ABI version, Strategy language, runtime adapter id/version, rules/spec compatibility, engine version, and supported capabilities.
- [x] **ABI-05**: Runtime ABI tests cover invalid output, thrown exception, timeout, forbidden capability, oversized output, malformed IPC, process exit/signal, and system failure handling.
- [x] **ABI-06**: Runtime ABI documentation states that Strategy execution cannot access system time, randomness, filesystem, network, environment, shell, database, or live model inference.

### Golden Parity Harness

- [x] **PAR-01**: Developer can run golden parity fixtures for engine outcomes, Chronicle projection, scoring, MatchSet summaries, analytics summaries, replay deep links, exports, runtime failures, privacy redaction, and deterministic ordering.
- [x] **PAR-02**: Golden fixtures are consumable by TypeScript and by future Go/non-JS runtime tests without relying on private implementation details.
- [x] **PAR-03**: Golden parity compares parsed canonical values by default and uses raw-byte/hash comparisons only where serialization or content hashes are explicit contract behavior.
- [x] **PAR-04**: Golden privacy fixtures fail if public outputs expose Strategy source, StrategyMemory, SoldierMemory, objective payloads, owner debug, raw Awareness Grid, stack traces, or private runtime internals.
- [x] **PAR-05**: Golden runtime fixtures distinguish Strategy failure from system failure and preserve deterministic ordering for repeated identical inputs.
- [x] **PAR-06**: Existing v1.6 analytics/replay/export evidence remains behaviorally unchanged through the golden fixture suite.

### Runtime Adapter Registry

- [x] **REG-01**: Developer can inspect a registry of supported Strategy languages and runtime adapters with ids, versions, capabilities, limits, production readiness, and isolation notes.
- [x] **REG-02**: Strategy Revisions carry immutable language/runtime adapter metadata as first-class fields while preserving existing JS/TS revision behavior.
- [x] **REG-03**: MatchSet, analytics, and gauntlet compatibility keys account for language, runtime adapter id, adapter version, ABI version, rule version, engine version, and Chronicle version.
- [x] **REG-04**: Compatibility checks fail closed with concrete mismatch reasons when adapter/language metadata differs in behavior-significant ways.
- [x] **REG-05**: JS/TS remains the only fully enabled runtime unless an adapter is explicitly marked experimental.

### One Non-JS Runtime Spike

- [x] **SPIKE-01**: Developer can run one experimental non-JS Strategy runtime adapter through the same ABI used by JS/TS subprocess-style execution.
- [x] **SPIKE-02**: The non-JS spike passes tiny valid-output, invalid-output, timeout, thrown-exception, stdout/stderr cap, and metadata fixture tests.
- [x] **SPIKE-03**: The non-JS spike is clearly marked experimental and does not imply production hostile-code isolation.
- [x] **SPIKE-04**: The non-JS spike preserves public runtime failure privacy and does not expose Strategy source, memories, objectives, stack traces, or private internals by default.

### Go Backend Spike

- [x] **GO-01**: Developer can run a minimal Go service that implements health and at least one read-only endpoint from the frozen service contract.
- [x] **GO-02**: The Go service consumes the same golden DTO fixtures or schema-compatible JSON as the TypeScript service boundary.
- [x] **GO-03**: The web app can integrate with the Go spike for a narrow read-only path without moving orchestration, job claiming, writes, or Strategy execution.
- [x] **GO-04**: Go spike tests prove DTO parity, deterministic ordering, privacy redaction, and graceful error handling for the selected endpoint.
- [x] **GO-05**: Documentation explains what the Go spike proves and explicitly lists backend responsibilities that remain in TypeScript.

## Future Requirements

Deferred to later milestones.

### Backend Migration

- **BACKX-01**: Developer can move orchestration, writes, jobs, or MatchSet execution behind a Go backend after read-only contract parity is proven.
- **BACKX-02**: Developer can generate or maintain a full OpenAPI document for the complete public and owner API after endpoint contracts stabilize.

### Runtime Expansion

- **RUNX-01**: User can submit production-supported non-JS Strategy Revisions after sandbox, package, Workshop, and compatibility semantics are proven.
- **RUNX-02**: Developer can promote a container, microVM, or WASM/WASI boundary to production-grade hostile-code isolation.
- **RUNX-03**: User can manage language package dependencies once package metadata, lockfile behavior, reproducibility, and security policy are designed.

### Product Surfaces

- **PRODX-01**: User can choose Strategy language in the public Workshop once more than JS/TS is production-supported.
- **PRODX-02**: User can participate in official scheduled tournaments or durable ratings after backend/runtime trust boundaries are production-ready.

## Out of Scope

Explicitly excluded from v1.7. Documented to prevent scope creep.

| Feature | Reason |
| --- | --- |
| Full Go backend rewrite | v1.7 proves the service boundary and a read-only Go slice before migrating orchestration or writes. |
| Moving job orchestration to Go | Match execution, job claiming, retries, and persistence workflows remain TypeScript until contract parity is proven. |
| Production multi-language Strategy support | v1.7 defines the ABI and adds one experimental spike, not a user-facing production language ecosystem. |
| Production-grade hostile-code sandbox replacement | Existing runtime boundaries stay mandatory; sandbox promotion needs a separate milestone. |
| Package dependency management for non-JS Strategies | Reproducible package metadata and lockfile semantics are larger than the first ABI spike. |
| Browser/API Strategy execution | Violates the core isolation rule; Strategy code continues to run only behind worker/runtime boundaries. |
| Durable ratings or official tournament operations | Boundary stabilization does not expand competition product commitments. |
| Custom or randomized arenas | Arena changes would distort parity fixtures and compatibility analysis. |
| Live model inference or human control during Matches | Contradicts the deterministic autonomous Match contract. |
