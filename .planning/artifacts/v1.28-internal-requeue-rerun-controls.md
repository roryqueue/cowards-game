## Recovery Semantics

Recovery requires an active quarantine row, a terminal `failed_system` job and Match, and no existing Chronicle. When applied, recovery releases quarantine, queues the job, clears terminal Match failure fields, refreshes MatchSet state, and grants one additional operator-authorized attempt by setting `max_attempts` to at least `attempts + 1`.

Repeated requests with the same idempotency key return a duplicate result and do not mutate execution state again.

## Rejection Guards

- Existing Chronicle.
- Inactive quarantine.
- Non-terminal job or Match state.
- `stale_artifact`.
- `malformed_runtime_result`.
- Missing or invalid operator action input.
