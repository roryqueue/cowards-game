# v1.18 Promotion Decision

**Milestone:** v1.18 Runtime Isolation and Multi-Language Exhibition Beta  
**Decision:** `promote-python-non-counted-exhibition-beta`  
**Runtime isolation status:** `readiness-evidence-only`

## Promoted

Python can be represented as an account-owned immutable Strategy Revision and used in non-counted exhibition beta MatchSets through the Strategy Execution Service / Runtime Broker ABI.

## Not Promoted

- Python ranked, ladder, counted, or gauntlet eligibility.
- Arbitrary PyPI/package installs or declared dependency resolution.
- Python backend, route, persistence, job lifecycle, Match completion, scoring, or public evidence ownership.
- Production sandbox certification.
- Silent fallback from stopped runtime-service or stopped Python runtime to JS/TS, Go, or web/API execution.
- Replacement or regression of JS/TS Strategy support.

## Evidence

- Hardened Python launch uses no shell, empty environment, and Python isolated mode where practical.
- Python validation uses a Python AST/compile host and public-safe diagnostic mapping.
- Boundary monitors fail on v1.18 registry/baseline/proof drift, Python execution outside runtime boundaries, backend ownership creep, privacy leaks, and premature counted eligibility.
- Signed-in local proof created JS/TS and Python account revisions, ran a non-counted exhibition through Go -> runtime-service -> runtime implementation, and opened replay evidence with zero runtime violations and no public private-data markers.

## Caveat

The runtime isolation evidence is stronger than v1.17 but remains local readiness evidence. It should not be described as production-grade hostile-code sandbox certification.
