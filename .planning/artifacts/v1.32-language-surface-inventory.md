# v1.32 Language Surface Inventory

**Milestone:** v1.32 Four-Language Production Strategy Support  
**Phase:** 222 - Language Surface Inventory  
**Date:** 2026-05-31  
**Status:** Phase 222 inventory complete; no behavior changed.

## Executive Answer

Python, Rust, and Zig can only be honestly promoted to fully supported counted Strategy languages when the active source of truth says they are supported, the runtime/provider boundary validates and executes them without JS/TS fallback, product surfaces derive labels and eligibility from that source, conformance and signed-in proofs cover all four languages, and public evidence stays source-free, memory-free, objective-free, diagnostics-free, and board-realistic.

The current codebase already has real multi-language validation and execution paths, but the active product truth is still split:

- `packages/spec/src/runtime.ts` is the strongest current source-of-truth candidate, but it still encodes Python/Rust/Zig as non-counted beta/experimental.
- `packages/spec/src/runtime-execution-service.ts` preserves the Runtime Broker boundary, but its implementation label is still JS/TS-oriented.
- Product code in Workshop, Account, exhibitions, MatchSet result, public strategy pages, and Learn/docs contains direct language-id branches and beta copy.
- `scripts/check-boundary-monitors.ts` actively enforces old non-promotion assumptions and must be converted into positive provider/parity/boundary monitors later.
- Historical artifacts from v1.18-v1.31 are useful baseline evidence and should not be rewritten unless an active monitor consumes them.

## Promotion Gate Inventory

Before Python, Rust, and Zig can be counted:

1. **Registry/provider truth:** one canonical record per language with provider id, source format, runtime target, adapter, public labels, docs refs, package policy, artifact policy, limits, privacy policy, and counted/entry eligibility.
2. **Provider contract:** validation, build/compile, artifact metadata, execution compatibility, diagnostics, and evidence are schema-validated and versioned.
3. **Execution boundary:** Python remains outside web/API/Go; Rust/Zig execute only immutable WASM/WASI artifacts through runtime-service / Runtime Broker / provider boundaries.
4. **Failure taxonomy:** invalid output, timeout, oversized output, forbidden capability, malformed runtime result, unavailable runtime, stale artifact, and system failure are distinguishable and public-safe.
5. **Parity matrix:** same-language and cross-language Match/MatchSet corpus proves deterministic behavior and result/replay parity across JS/TS, Python, Rust, and Zig.
6. **Product unification:** Workshop, Account, competition entry, result, replay, public discovery, public strategy pages, and Learn/docs all derive labels and eligibility from providers.
7. **Privacy proof:** public evidence scans prove no Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw diagnostics, host paths, environment values, package paths, tokens, DB details, quarantine details, or operator internals leak by default.
8. **Board realism proof:** replay/result browser proof must show plausible full Match starts with Soldiers/terrain inside declared board bounds.
9. **Monitor conversion:** old non-promotion monitors must become positive monitors for provider completeness, eligibility parity, no fallback, ABI/versioning, privacy, and approved execution boundaries.
