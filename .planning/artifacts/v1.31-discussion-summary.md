# v1.31 Discussion Summary

**Milestone:** v1.31 Public Site Spine and Discovery Reads
**Discussed:** 2026-05-31
**Mode:** Compact milestone-wide discussion

## Confirmed Backbone

- Use a new public discovery namespace. Discovery DTOs/routes are separate from `match-execution-app-v1`.
- Discovery reads are server-first, typed, public/account-safe, link-rich, and compositional.
- Discovery reads may summarize existing public-safe projections and canonical hrefs, but must not add fields to existing public execution DTOs.
- Empty and unavailable states are first-class public states.
- Public discovery must not inspect Strategy source, owner debug data, raw Chronicle private data, runtime internals, quarantine/recovery/operator data, or execute Strategy code.
- Use a restrained competitive site shell with Home, Watch, Competitions, Learn, Workshop, and Account.
- Home is a real discovery dashboard, not a marketing splash.
- Watch is a denser evidence feed.
- `/competitions` is the canonical public competition directory; existing `/ladder/[seasonId]` and `/exhibitions/new` remain compatible specific routes.
- `/competitions/[competitionId]/enter` is the signed-in entry spine and should reuse existing creation behavior where possible.
- Cross-links are evidence-oriented and contextual, not a generic link farm.
- Safety includes schema/DTO tests, rendered-page scans, proof artifact scans, and boundary monitors.
- Browser proof may use seeded/fixture public-safe data and must label fixture-backed versus live coverage.
- Final audit must explicitly preserve all non-claims: frozen execution contract, no execution DTO drift, no execution/runtime/retry/recovery/quarantine/job/scoring/Chronicle changes, no Strategy execution in web/API/Go, JS/TS counted only, Python/Rust/Zig non-counted exhibition beta only, Preview 1 stdin/stdout JSON active.

## Phase Carry-Forward

- **Phase 212:** Define discovery contracts before UI.
