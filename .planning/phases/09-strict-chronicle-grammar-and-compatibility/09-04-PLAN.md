# Plan 09-04: Snapshot Boundary Grammar

## Objective

Create a pure snapshot boundary validator for Chronicle boundary snapshots.

Purpose: This implements GRAM-04 and the missing-sequence portion of GRAM-05/GRAM-08 while honoring D-01, D-02, D-03, D-04, and D-10. It validates what the Chronicle claims at snapshot boundaries without re-running Strategy source.

Output: A tested `packages/replay/src/snapshot-boundaries.ts` module that Plan 09-05 can wire into `validateChronicle`.

## Tasks

<task type="auto" tdd="true">
  <name>Task 1: Specify snapshot kind to event type rules</name>
  <files>packages/replay/src/snapshot-boundaries.test.ts, packages/replay/src/snapshot-boundaries.ts</files>
  <behavior>
    - `MATCH_START` attaches to `MATCH_STARTED`.
    - `ROUND_START` attaches to `ROUND_STARTED`.
    - `ROUND_END` attaches to the last event in that Round before the next Round, a Contraction boundary, or `MATCH_ENDED`.
    - `ACTIVATION_START` attaches to `ACTIVATION_STARTED`.
    - `ACTIVATION_END` attaches to the last event in that Activation before the next Activation, the Round end boundary, or `MATCH_ENDED`.
    - `CONTRACTION` attaches to `CONTRACTION_RESOLVED`.
    - `MATCH_END` and `TERMINAL` attach to `MATCH_ENDED`.
    - A snapshot sequence that references no event fails with `SNAPSHOT_MISSING`.
    - A snapshot kind attached to the wrong event type or wrong end-boundary sequence fails with `SNAPSHOT_BOUNDARY_INVALID`.
  </behavior>
  <action>
    Write failing table-driven tests that mutate snapshot `kind` and `sequence` values on legal Chronicles built through `buildChronicleFromMatch`. Include explicit cases where `ROUND_END` is moved to a mid-round event with the same `roundNumber`, and `ACTIVATION_END` is moved to a mid-activation event with the same activation context; both must fail with `SNAPSHOT_BOUNDARY_INVALID`. Follow the existing local helper style in `validate.test.ts`; do not import `@cowards/test-utils` into `@cowards/replay` because `@cowards/test-utils` already depends on replay. Then create `snapshot-boundaries.ts` with an explicit snapshot-kind-to-event-type matrix plus end-boundary sequence detection for Round and Activation end snapshots.

    Keep this module about boundary grammar only. Board transition comparison belongs to Plan 09-05.
  </action>
  <verify>
    <automated>pnpm --filter @cowards/replay test -- snapshot-boundaries.test.ts</automated>
  </verify>
  <done>Snapshot kind/event-type mismatches and missing referenced events fail with stable errors.</done>
</task>

<task type="auto" tdd="true">
  <name>Task 2: Enforce boundary context consistency</name>
  <files>packages/replay/src/snapshot-boundaries.ts, packages/replay/src/snapshot-boundaries.test.ts</files>
  <behavior>
    - `ROUND_START` and `ROUND_END` snapshots carry the same `roundNumber` as the referenced boundary event.
    - `ACTIVATION_START` and `ACTIVATION_END` snapshots carry the same `activationId`, `activationIndex`, `roundNumber`, `actingPlayerId`, and `soldierId` as the referenced boundary event where present.
    - Matching context alone is insufficient for `ROUND_END` and `ACTIVATION_END`; the snapshot sequence must also be the computed end boundary for that Round or Activation.
    - `CONTRACTION` snapshots carry the same Round/Phase context as `CONTRACTION_RESOLVED` when present.
    - `MATCH_END` and `TERMINAL` snapshots have an `outcome` matching the terminal event payload.
  </behavior>
  <action>
    Add context comparison helpers for boundary snapshots and referenced events. Add bounded sequence helpers that compute the final event sequence for the snapshot's Round or Activation from Chronicle event context only. Return `CONTEXT_MISMATCH` for context contradictions and `SNAPSHOT_BOUNDARY_INVALID` for boundary-kind or wrong end-boundary sequence errors.

    Do not infer hidden state from engine execution; compare only fields present in the Chronicle and boundary snapshot.
  </action>

## Threat Model

| Threat ID | Category | Component | Disposition | Mitigation Plan |
| --- | --- | --- | --- | --- |
| T-09-04-01 | Tampering | snapshot boundary references | mitigate | Validate every snapshot sequence references an existing compatible event. |
| T-09-04-02 | Tampering | snapshot context | mitigate | Compare snapshot context against referenced boundary event context and terminal outcome payloads. |
| T-09-04-03 | Repudiation | snapshot failures | mitigate | Use stable `SNAPSHOT_BOUNDARY_INVALID`, `SNAPSHOT_MISSING`, and `CONTEXT_MISMATCH` codes with clear messages. |
