# Phase 133: Executable Container Runtime Candidate Lane - Context

**Gathered:** 2026-05-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 133 turns the selected Docker/runc container subprocess lane into real executable candidate evidence. It updates the v1.19-oriented sandbox evaluation so v1.20 can run the actual adapter, fail loudly on missing controls or substituted evidence, and record honest readiness evidence without claiming production sandbox certification.

</domain>

<decisions>
## Implementation Decisions

### Container Candidate Execution
- **D-01:** The Docker/container subprocess lane is the primary executable stronger sandbox candidate for v1.20.
- **D-02:** When Docker and the configured image are available, the candidate lane must execute real probes through the `container-subprocess` adapter.
- **D-03:** Strict candidate commands must fail loudly when Docker is unavailable, the image is missing, the daemon fails, controls are unsupported, the adapter fails, evidence is stale, or another candidate is silently substituted.
- **D-04:** Missing Docker/image/control evidence is a setup or non-promotion record, not a silent skip.

### Controls To Prove
- **D-05:** Evidence must cover network denial, read-only root, tmpfs scratch, dropped capabilities, no-new-privileges, PID/memory/CPU controls, no shell, and strict IPC.
- **D-06:** Adapter launch uses structured argv with `shell: false`; no proof path should introduce shell interpolation for hostile Strategy source.
- **D-07:** Failure taxonomy must distinguish Docker unavailable, image unavailable, daemon failure, unsupported control, adapter failure, Strategy violation, timeout, and system failure.

### Artifact Claims
- **D-08:** Artifacts must be v1.20-specific and compare subprocess baseline with the container candidate.
- **D-09:** Evidence wording must say what the container candidate proves and does not prove; no production sandbox claim is allowed from Docker/runc alone unless future evidence genuinely supports it.
- **D-10:** `runsc` remains fail-loud only; do not make gVisor pass because Docker/runc passed.

### the agent's Discretion
The agent may decide whether the v1.20 candidate evidence is generated by extending the existing evaluator or by adding a narrow companion script, but monitor/check commands must stay easy to run from `package.json`.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- The current adapter already requests `--network none`, `--read-only`, tmpfs `/tmp`, memory/CPU/PID limits, `--cap-drop ALL`, and `--security-opt no-new-privileges`.
- Existing tests verify Docker args, timeout mapping, launch failure mapping, stdio caps, and unsafe image rejection.
- The current evaluator has `container-subprocess` as optional-executable and gates live execution behind `COWARDS_RUN_CONTAINER_SANDBOX=1`.

### Known Gaps
- Current evaluation schema and readiness naming are v1.19-oriented.
- Container evidence is not yet the required v1.20 lane when Docker is available.
- `runsc` strict checks currently fail because no executable adapter exists and `runsc` is unavailable locally.

### Integration Points
- Phase 134 consumes the candidate lane for hostile probe parity.
- Phase 138 consumes the strict candidate evidence for signed-in proof.
- Phase 139 consumes the artifact for promotion/non-promotion decisions.

</code_context>

<specifics>
## Specific Ideas

Prefer a strict command that makes setup failures unmistakable: "Docker available but candidate evidence missing" should be different from "Docker not installed" and from "Strategy caused runtime violation."

</specifics>

- Production sandbox certification.

</deferred>
