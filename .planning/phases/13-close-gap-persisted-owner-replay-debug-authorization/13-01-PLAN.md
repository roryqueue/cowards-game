# Plan 13-01: Persisted Owner Replay Authorization

## Objective

Make persisted Match replay pages capable of returning owner-debug DTOs only after server-side authorization confirms the requested owner is a participant in the persisted Match.

## Read First

- `.planning/phases/13-close-gap-persisted-owner-replay-debug-authorization/13-CONTEXT.md`
- `.planning/phases/13-close-gap-persisted-owner-replay-debug-authorization/13-RESEARCH.md`
- `apps/web/app/matches/server.ts`
- `apps/web/app/matches/replay-ready.ts`
- `apps/web/app/matches/server.test.ts`

## Tasks

1. Extend `buildReadyReplayFromStoredChronicle` so callers may pass an authorized requested-owner list into `trustedOwnerReplayOptions`.
2. Add a small owner authorization dependency seam to `createMatchReplayServer`.
3. In the default persisted replay path, authorize requested owner debug only through trusted server data. Query params remain advisory and must downgrade to public if authorization fails.
4. Add tests proving:
   - public persisted replay remains public and privacy-safe by default;
   - query-requested owner ids are not trusted without the resolver;
   - authorized requested owners upgrade to owner mode and receive `ownerDebug.soldierInactivityExplanations`;
   - nonparticipants, missing owner ids, and wrong owners stay public.

## Threat Model

- `ownerPlayerId` query guessing must not expose owner data.
- Explicit trusted server calls remain available for tests/fixtures, but browser request params alone do not establish trust.
- React must not infer owner-debug causes; the server returns DTOs from replay/engine-derived helpers.

## Verification

- `pnpm --filter @cowards/web test -- server.test.ts owner-debug.test.ts`

## Done

Persisted owner replay mode is reachable for authorized owners, unreachable for public/unauthorized requests, and covered by focused server tests.
