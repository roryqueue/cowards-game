# v1.25 Milestone Discussion Context

**Gathered:** 2026-05-30
**Status:** Approved for phase planning

## Milestone Boundary

v1.25 freezes the app-facing Match execution boundary so execution internals and app/result/replay UX can proceed in parallel. This is a contract milestone, not a runtime promotion milestone.

## Approved Milestone Decisions

- JS/TS remains the counted Strategy path.
- Python, Rust, and Zig remain non-counted exhibition beta.
- Preview 1 stdin/stdout JSON remains the active WASM/WASI execution ABI.
- No production sandbox certification, Component Model/WIT migration, direct-export ABI migration, or broad ranking/ladder expansion is in scope.
- Go owns orchestration, persistence-facing backend behavior, Match lifecycle, scoring, public evidence, retry policy, and promotion decisions.
- Runtime-service owns hostile Strategy execution only through schema-validated ABI envelopes and registered runtime implementations.
- Strategy code must not execute in web/API/Go.
- Public output must not expose Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw diagnostics, host paths, env values, tokens, DB details, package paths, or private runtime internals by default.

## Approved Lifecycle Model

Retryability is not a top-level lifecycle state. It is a field on lifecycle/failure evidence.

The app-facing lifecycle contract should model:

- `state`: `queued`, `accepted`, `running`, `complete`, `failed`, `degraded`, or `unavailable`.
- `terminal`: boolean.
- `retryDisposition`: `retryable`, `non_retryable`, or `not_applicable`.
- `failureCategory`: optional public category such as `strategy_failure`, `system_failure`, `timeout`, `runtime_unavailable`, `malformed_runtime_result`, `stale_artifact`, `blocked`, `missing_chronicle`, or `no_result`.
- `resultAvailability`: `none`, `partial`, or `complete`.
- `replayAvailability`: `none`, `pending`, `available`, `stale`, or `missing`.

`degraded` is a terminal public-evidence state with partial trustworthy evidence. It is not a clean counted outcome by default. Strategy failure must remain distinct from system failure; system failure, timeout, malformed runtime result, missing Chronicle, stale artifact, and unavailable runtime must never be disguised as game outcome.

## Approved DTO and Evidence Boundaries

Use three layers:

1. Public app DTOs consumed by result/replay pages and public routes.
2. Owner/test-only DTOs for explicitly gated debug/detail surfaces.
3. Execution-internal envelopes for runtime-service and Go orchestration internals.

The app-facing v1 DTO family should include:

- `LifecycleV1`
- `MatchSetSummaryV1`
- `MatchResultV1`
- `ReplayMetadataV1`
- `ReplayEvidenceV1`
- `RuntimeEvidenceV1`
- `FailureEvidenceV1`

Public DTOs may expose Match/MatchSet ids, Strategy Revision ids, display/runtime labels, lifecycle state, public failure category, stable public message/copy key, retry disposition, replay availability, scoring/result summary where trustworthy, redacted runtime label/eligibility evidence, and public Chronicle/replay projection.

Unknown, stale, malformed, or unversioned DTO payloads must fail closed at app adapters and contract tests. UI may show public-safe unavailable/failed/degraded copy, but it must not best-effort parse private or execution-shaped data.
