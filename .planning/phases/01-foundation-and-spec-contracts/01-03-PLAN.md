# Plan 01-03: Canonical Spec Contracts and Fixtures

<objective>
Implement the Phase 1 `packages/spec` contract layer with canonical types, constants, schemas, compatibility versions, and fixtures.
</objective>

<requirements>
- FOUND-05: The project records engine, runtime, spec, Chronicle, Strategy Revision, and Arena Variant versions wherever Match reproducibility depends on them.
- SPEC-01: Developer can import canonical TypeScript types.
- SPEC-02: Developer can validate all runtime inputs and outputs with Zod schemas.
- SPEC-03: Developer can rely on canonical constants.
- SPEC-04: Developer can use canonical discriminated unions.
- SPEC-05: Developer can generate valid test fixtures.
</requirements>

<must_haves>
- `packages/spec` includes contracts plus fixtures.
- Valid and invalid fixtures are separated.
- `COMPATIBILITY_VERSIONS` has core six fields only.
- Spec version is `1.0.0`; implementation artifacts are `0.1.0`.
- No engine transition behavior is implemented.
</must_haves>

<truths>
- D-03: `packages/spec` owns canonical TypeScript types, constants, Zod schemas, event/action/status unions, version IDs, and canonical fixtures.
- D-04: `packages/spec` must not depend on any internal workspace package.
- D-16: `packages/spec` provides both named canonical fixtures and validated builder functions.
- D-17: Fixture builders separate valid and invalid namespaces.
- D-18: Valid fixture builders default to legal states.
- D-19: Invalid fixtures are explicit and reserved for negative tests.
- D-20: Phase 1 fixtures include starting board, Soldiers, constants, Arena Variants, runtime input/output examples, seed values, and named board setups for blocked movement, push, Backstab, off-board fall, contraction, and no-advance stoning.
- D-21: Represent compatibility versions as a structured `CompatibilityVersions` object.
- D-22: Initial version values use spec `1.0.0` and implementation artifacts `0.1.0`.
- D-23: Include the core six fields in the compatibility object: `spec`, `engine`, `runtimeJs`, `chronicle`, `strategyRevision`, and `arenaVariant`.
- D-24: Do not include `webApp`, `worker`, `databaseSchema`, or `apiContract` versions in Phase 1.
</truths>

<threat_model>
| ID | Threat | Severity | Mitigation |
|----|--------|----------|------------|
| T-1-05 | Runtime boundary later trusts unvalidated strategy output. | high | Define Zod schemas for StrategyResult, SoldierBrainResult, Actions, and runtime inputs in `packages/spec`. |
| T-1-06 | Test fixtures accidentally normalize illegal states. | medium | Separate `fixtures.valid.*` from `fixtures.invalid.*` and validate legal builder outputs. |
</threat_model>
