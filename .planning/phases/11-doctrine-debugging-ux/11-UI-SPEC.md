| Property | Value |

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Label-to-control gaps, compact metadata rows, Awareness Grid cell internals |
| sm | 8px | Validation row padding, match row gaps, chip text padding, control groups |
| md | 16px | Default panel padding, Workshop/replay grid gap, page padding on desktop |
| lg | 24px | Large panel group spacing and expanded empty-state padding |
| xl | 32px | Major vertical breaks only when a phase view has no adjacent panel |
| 2xl | 48px | Empty-state maximum top spacing; do not use inside dense panels |
| 3xl | 64px | Reserved for future full-page states; not used in Phase 11 panels |

Exceptions: existing desktop buttons keep `min-height: 36px`; mobile/full-width controls must provide at least a 44px touch target through padding or layout. Replay board and Awareness Grid keep fixed-format aspect/track sizing from the existing CSS.

## Typography

Use only these sizes for new Phase 11 UI additions:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.45 |
| Label / metadata | 12px | 700 | 1.3 |
| Section heading | 18px | 700 | 1.25 |
| Page title / position | 22px | 700 | 1.2 |

Weight contract: use only 400 and 700 for new additions. Existing CSS may keep its current 650/750 declarations until a broader cleanup, but Phase 11 should not introduce additional weights.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#f4f6f3` | Page background and negative space around work surfaces |
| Secondary (30%) | `#ffffff` / `#e8ece6` | Panels use white; validation rows, chips, and inactive cell surfaces use secondary green-gray |
| Accent (10%) | `#256d85` | Primary buttons, active sample row border, replay links, focus outline, selected replay/Soldier affordances |
| Destructive | `#b42318` | Validation errors, invalid runtime output, thrown exception labels, failed system copy |
| Warning | `#9a6b12` | Pending/running/replay-not-ready explanations and intentional failure-mode sample tags |
| Success | `#2f7d46` | Valid draft, available replay, successful sample tags |

Accent reserved for: `Validate source` primary state, `Submit revision` primary state, `Open replay` links, active sample rows, selected timeline/Soldier states, focus-visible outlines, and owner-mode status chip text. Do not use accent for every link-like metadata value.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Validate source` |
| Secondary CTAs | `Submit revision`, `Launch test`, `Open replay`, `Refresh status` |
| Empty state heading | `No validation issues` |
| Empty state body | `This draft passes the Strategy API checks. Submit a revision or launch a Workshop test to inspect runtime behavior.` |
| Error state | `Replay unavailable: this Match does not have a stored Chronicle yet. Refresh status after the Match completes.` |
| Destructive confirmation | None in scope. Keep the existing replace-draft confirmation: `Replace draft: this will overwrite the current unsaved source with the selected template.` |

Copy rules:

- Validation/runtime messages must name one Strategy API constraint and one remediation step.
- Keep row copy to a label, one concise explanation sentence, and one remediation sentence.
- Do not dump docs, stack traces, raw Strategy source, StrategyMemory, SoldierMemory, or objective payloads in UI copy.
- Use canonical terms exactly: Soldier, Match, Phase, Round, Activation, Cycle, Action, Advance, STONE, FALLEN, Chronicle.

## Phase Surfaces

## Visual Priority

| Surface | Primary Focal Point | Secondary Priority |
|---------|---------------------|--------------------|
| Workshop | Editor and validation guidance column. | Sample catalog supports source selection; submit/test panels remain operational controls, not the visual center. |
| Replay | Board and timeline in the center column. | Selected-Soldier owner explanation is secondary in the inspector and must not compete with board readability. |
| Public replay | Board, timeline, and public inspector. | Owner debug affordances must be absent, not merely visually de-emphasized. |

| Surface | Contract |
|---------|----------|
| Workshop validation guidance | Replace single-message validation rows with structured guidance rows. Each row shows severity/code, Strategy API constraint, current issue message, and one remediation step. Rows stay in the existing `workshop-validation-panel` and use the current `validation-list` / `validation-row` visual language. |
| Sample Strategy catalog | Expand the existing Templates panel into a sample catalog without adding a marketing treatment. Rows remain compact buttons. Each row shows sample name, a one-line purpose, and small tags for `Pattern`, `Failure mode`, or `Valid`. Loading a sample still replaces the editor source only after the existing dirty-draft confirmation. |
| Workshop replay availability | Match result rows continue to show `Open replay` only when `canOpenReplay(match)` is true. All other states show a reason, not a dead link. |
| Replay unavailable page | Keep the existing `Replay unavailable` shell. The message must include the reason and next step, not just a generic failure. |
| Owner replay explanation | In owner/debug mode, the right inspector adds a selected-Soldier explanation section below Soldier facts and above raw advanced debug JSON. The section renders DTO fields only. |
| Public replay | Public mode keeps the existing clean replay: board, timeline, public event inspector, Soldier facts, and `Public view` status chip. No owner explanation, Awareness Grid, raw ownerPrivate JSON, Strategy source, memory, objective payload, or raw runtime details render in public mode. |

## Workshop Validation Guidance Rows

Validation row anatomy:

| Element | Required Rendering |
|---------|--------------------|
| Code line | `ERROR / {CODE}` or `WARNING / {CODE}` in monospace-compatible label sizing |
| Constraint line | `Constraint: {Strategy API constraint}` |
| Explanation line | Existing or mapped issue message, rewritten only if the helper owns the copy |
| Remediation line | `Next: {single remediation step}` |

Required constraint/remediation mappings:

| Code / Type | Constraint Copy | Remediation Copy |
|-------------|-----------------|------------------|
| `SOURCE_TOO_LARGE` | `Strategy source must stay within the source byte limit.` | `Remove unused helper code or comments before validating again.` |
| `FORBIDDEN_PATTERN` | `Strategy source cannot use host capabilities.` | `Remove the forbidden capability and keep all decisions inside the Strategy input/output API.` |
| `IMPORT_NOT_ALLOWED` | `Strategy source must be self-contained.` | `Inline deterministic helper code instead of importing modules.` |
| `MISSING_DEFAULT_EXPORT` | `Strategy source must export one default Strategy object.` | `Add export default with selectActivations and soldierBrain methods.` |
| `MISSING_SELECT_ACTIVATIONS` | `Strategy API requires selectActivations(input).` | `Return activationOrders and strategyMemory from selectActivations.` |
| `MISSING_SOLDIER_BRAIN` | `Strategy API requires soldierBrain(input).` | `Return one Action and soldierMemory from soldierBrain.` |
| `ASYNC_METHOD_NOT_ALLOWED` | `Strategy methods must return synchronously.` | `Remove async/await and return the Action directly.` |
| `TRANSPILE_FAILED` | `Strategy source must parse as supported JS/TS.` | `Fix the syntax error, then validate source again.` |
| `ENGINE_INCOMPATIBLE` | `Strategy compatibility must match the current engine/spec versions.` | `Revalidate against the current Workshop runtime before submitting.` |
| `INVALID_OUTPUT` | `Strategy runtime output must match the Action/result schema.` | `Return only schema-valid activation orders, Action, StrategyMemory, and SoldierMemory.` |
| `TIMEOUT` | `Strategy methods must finish within the runtime time limit.` | `Remove unbounded loops and expensive computation.` |
| `THROWN_EXCEPTION` | `Strategy methods must not throw during evaluation.` | `Guard nullable input and return a fallback Action.` |
| `FORBIDDEN_CAPABILITY` | `Strategy runtime cannot access host capabilities.` | `Remove filesystem, network, process, timer, random, or environment access.` |
| `OVERSIZED_OUTPUT` | `Strategy runtime output must stay within the output byte limit.` | `Store only compact serializable memory and return a small Action payload.` |

If the runtime sends a code not listed above, render:

- Constraint: `Strategy API constraint unavailable for {CODE}.`
- Next: `Review the issue message and compare the draft with the sample Strategies.`

## Sample Strategy Catalog

Presentation contract:

| Requirement | Contract |
|-------------|----------|
| Panel title | Rename or supplement `Templates` as `Sample Strategies`. |
| Row density | Each sample row is one button-like row with title, one-line purpose, and up to three chips. No nested cards. |
| Categories | `Movement`, `Push`, `Backstab`, `Stone`, `Runtime violation`, `Invalid output`, `Do nothing`. |
| Valid samples | Use success chip text `Valid sample`. |
| Failure samples | Use warning chip text `Failure mode`; do not use destructive color unless the selected sample currently fails validation. |
| Source loading | Preserve current dirty-draft confirmation before replacing editor content. |
| Test reuse | Sample IDs must be stable and suitable for fixture/test references. |

Required sample rows:

| Sample | Row Purpose | Tags |
|--------|-------------|------|
| Basic advance and turn | `Advance when clear, otherwise turn to keep facing useful space.` | `Movement`, `Valid sample` |
| Push setup | `Demonstrates positioning for Push resolution.` | `Push`, `Valid sample` |
| Backstab setup | `Shows facing and adjacency needed for Backstab.` | `Backstab`, `Valid sample` |
| Stone and blocking | `Uses STONE to create blocking pressure.` | `Stone`, `Valid sample` |
| Runtime timeout | `Demonstrates a Strategy that exceeds the runtime limit.` | `Runtime violation`, `Failure mode` |
| Thrown exception | `Demonstrates a Strategy exception path.` | `Runtime violation`, `Failure mode` |
| Invalid output | `Returns an Action payload that fails schema validation.` | `Invalid output`, `Failure mode` |
| Do nothing | `Selects no useful Activations so Soldiers remain idle.` | `Do nothing`, `Failure mode` |

## Owner Replay Debug Explanations

Owner explanation placement:

- Render only when `canShowOwnerDebug(data)` is true and the owner debug toggle is enabled.
- Place in the right replay inspector, directly after selected Soldier facts.
- Label the section `Why this Soldier did nothing`.
- Show raw JSON only after this structured explanation, under the existing advanced debug panel.
- If no explanation exists for the selected Soldier/sequence, show `No inactivity explanation for this Soldier at this position.`

Required cause-code copy:

| Cause Code | Label | Detail Copy |
|------------|-------|-------------|
| `NOT_SELECTED` | `Not selected` | `This Soldier was not selected in the activation order for this Activation.` |
| `INVALID_ACTION` | `Invalid Action` | `The Strategy returned an Action that the runtime or engine rejected.` |
| `BLOCKED_MOVEMENT` | `Blocked Advance` | `The replay DTO reports that this Soldier's Advance was blocked.` |
| `TIMEOUT` | `Runtime timeout` | `The Strategy did not return before the runtime time limit.` |
| `THROWN_EXCEPTION` | `Strategy exception` | `The Strategy threw during evaluation. Public replay must not show raw stack details.` |
| `STONE` | `STONE` | `This Soldier was STONE and could not act.` |
| `FALLEN` | `FALLEN` | `This Soldier had fallen and could not act.` |
| `MATCH_ENDED` | `Match ended` | `The Match had reached a terminal outcome, so no further Actions were processed.` |

DTO rendering contract:

- React may render `code`, `label`, `detail`, `sourceEventSequence`, and public-safe event references.
- React must not infer legality from board position, movement rules, Backstab rules, Push rules, contraction rules, or Action schemas.
- Owner explanations must come from replay/engine/runtime-derived DTOs assembled before rendering.

## Public Replay Privacy

Public replay must not render:

- Strategy source.
- StrategyMemory.
- SoldierMemory.
- Objective payloads.
- Raw Awareness Grid details.
- Raw runtime exception stack, process output, or private violation details.
- `projection.ownerPrivate`.
- Owner-only inactivity explanation DTOs.

Public replay may render:

- Public board state.
- Public timeline entries and labels.
- Public runtime violation label if it is already part of the public Chronicle projection.
- Soldier status, owner, position, facing, and public event references already present in projection data.

## Responsive Layout

Workshop:

| Viewport | Contract |
|----------|----------|
| `>=1180px` | Keep current 3-column grid: `260px minmax(520px, 1fr) 340px`. Sample catalog left, editor/validation center, submit/test right. |
| `760px-1179px` | Keep two-column mode with editor/validation ordered first. Validation rows must wrap without horizontal scroll. |
| `<760px` | Keep one-column order: header, sample catalog, editor, validation, submit, revision history, test. Buttons may become full-width. |

Replay:

| Viewport | Contract |
|----------|----------|
| `>=1180px` | Keep current 3-column grid: metadata rail, board/timeline center, inspector rail. |
| `900px-1179px` | Center board spans first row; rails stack beneath in two columns. |
| `<900px` | Single column. Owner explanation follows selected Soldier facts and stays above raw debug JSON. |

All new rows must use wrapping text, `overflow-wrap: anywhere` for IDs/codes, and no viewport-width font scaling.

## Testable Selectors

Use semantic roles and labels first; add stable selectors for repeated diagnostic rows where text varies.

| Element | Selector Contract |
|---------|-------------------|
| Sample catalog | `data-testid="workshop-sample-catalog"` |
| Sample row | `data-testid="workshop-sample-row"` plus `data-sample-id="{stable id}"` |
| Validation guidance list | `data-testid="workshop-validation-guidance-list"` |
| Validation guidance row | `data-testid="workshop-validation-guidance-row"` plus `data-validation-code="{CODE}"` |
| Replay availability row | `data-testid="workshop-replay-availability"` plus `data-replay-state="available|unavailable"` |
| Replay unavailable page message | `data-testid="replay-unavailable-message"` |
| Owner debug toggle | `data-testid="replay-owner-debug-toggle"` |
| Soldier inactivity explanation | `data-testid="replay-soldier-inactivity-explanation"` plus `data-cause-code="{CAUSE_CODE}"` |
| Public replay shell | Keep `aria-label="Match replay"` and status chip text `Public view`; do not add owner-debug selectors in public mode. |

Selector rules:

- Do not build selectors from unsanitized Strategy source, raw Match IDs, memory payloads, or objective payloads.
- Cause codes and validation codes must remain uppercase stable enum values for tests.
- Keep existing accessible labels: `Matches`, `Replay timeline`, `Timeline events`, `Replay inspector`, `Soldiers`, and `Awareness Grid`.
