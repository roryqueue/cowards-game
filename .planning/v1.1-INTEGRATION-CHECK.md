## Detailed Findings

### BLOCKER: Persisted Owner Debug Explanations Are Not Reachable

**Affected requirements:** DEBUG-04, DEBUG-05
**Integration path expected:** Phase 11 debug DTOs -> persisted Chronicle replay -> owner replay route -> replay client owner overlay
**Status:** BROKEN for persisted Match Chronicles

Evidence:
- `ReplayPage` passes query-derived options from `resolveOwnerDebugReplayOptions(...)` into `getMatchReplay(...)`.
- The resolver only returns `{ allowOwnerDebug: true, requestedOwnerPlayerId }`; it does not produce trusted `{ mode: "owner", ownerPlayerId }`.
- `buildReadyReplayFromStoredChronicle(...)` calls `trustedOwnerReplayOptions(metadata, options)` with no authorized owner list, so query-requested owner IDs are intentionally downgraded to public.
- Existing server tests assert this behavior: query-requested persisted owner IDs remain public, and only direct trusted server calls with `mode: "owner"` return owner data.
- The fixture path separately passes an authorized owner list, so owner debug works in fixture E2E but not in real persisted replay pages.

Specific files:
- `apps/web/app/matches/[matchId]/replay/owner-debug.ts`
- `apps/web/app/matches/server.ts`
- `apps/web/app/matches/replay-ready.ts`
- `apps/web/app/matches/replay-fixture.ts`
- `apps/web/app/matches/server.test.ts`

Missing step:
- Persisted replay loading needs an authenticated/authorized owner resolver, or a trusted server-side owner path, before `ReplayReadyDto.ownerDebug.soldierInactivityExplanations` can reach the replay client for real Matches.

### WARNING: Runtime Failure Debug UX Is Not Service-Backed End-to-End

**Affected requirements:** ISO-07, DEBUG-04, DEBUG-06, REL-04
**Integration path expected:** hostile/failing Strategy source -> worker/runtime adapter -> Chronicle `RUNTIME_VIOLATION` -> persisted replay -> owner debug explanation -> public privacy check
**Status:** PARTIAL

Verified pieces:
- Runtime-js integration produces `RUNTIME_VIOLATION` and public projection omits raw exception detail.
- Worker tests keep gameplay runtime violations on the Match completion path and classify subprocess system failures separately.
- Replay fixture browser tests render runtime failure callouts and owner-only Soldier inactivity explanations without public leakage.

Gap:
- `apps/web/e2e/workshop-to-replay.spec.ts` verifies the service-backed happy path only. It does not select a failure-mode sample and assert runtime violation -> persisted replay -> owner debug explanation.

### WARNING: Compatibility Alias API Routes Are Orphaned

**Status:** WIRED for current routes, ORPHANED for aliases

Current Workshop UI calls `/api/workshop/revisions`, `/api/workshop/revisions/[revisionId]/source`, `/api/workshop/tests`, and `/api/workshop/tests/[matchSetId]`. Legacy aliases under `/api/workshop/submit`, `/api/workshop/source`, `/api/workshop/test`, and `/api/workshop/test/[matchSetId]` re-export or duplicate handlers but have no current in-repo consumer.

## Verified Wiring

### Generated Fixtures -> Strict Grammar -> Replay Viewer

Status: WIRED

- `@cowards/test-utils` exports canonical replay scenarios and IDs.
- `apps/web/app/matches/replay-fixture.ts` imports `getCanonicalReplayScenario`/`getCanonicalReplayScenarioIds` and builds route-safe replay fixture Match IDs.
- Fixture and persisted replay both converge at `buildReadyReplayFromChronicle(...)`.
- `buildReadyReplayFromChronicle(...)` calls `createReplay(...)` before projection; `createReplay(...)` calls `validateChronicle(...)`.
- `validateChronicle(...)` includes version, event order, required events, snapshots, semantic grammar, snapshot boundaries, transition contradiction checks, and hash validation.
- Playwright visual checks load the fixture catalog and open scenario replay hrefs, then screenshot generated scenario boards.

Requirements covered: FID-01, FID-02, FID-03, FID-04, FID-05, FID-06, FID-07, GRAM-01 through GRAM-05.

### Public Projection Privacy + Debug Overlay Gates

Status: WIRED for public privacy, BROKEN for persisted owner access

- `projectPublicChronicle(...)` strips private key names including `ownerDebug`, `soldierInactivityExplanations`, `strategyMemory`, `soldierMemory`, `objectivePayload`, raw runtime details, and Strategy source.
- `ReplayReadyDto.ownerDebug` is only assembled when mode is trusted owner.
- Replay client starts owner debug hidden and renders explanations only behind the owner debug checkbox.
- Fixture browser tests confirm public views hide owner debug and private runtime details.
- Persisted owner access is not wired from replay route to trusted owner mode; see BLOCKER.

Requirements covered: GRAM-07, DEBUG-06.
Requirements partially/broken: DEBUG-04, DEBUG-05.

### Runtime Adapter -> Worker -> Chronicle

Status: WIRED

- `@cowards/runtime-js/worker` exports `createRuntimeFromRevision`, worker-thread and subprocess adapters, adapter metadata, and subprocess failure types.
- Worker startup reads `STRATEGY_EXECUTION_ADAPTER`, logs adapter ID and isolation boundary, and fails closed for unknown IDs.
- Worker `loadRunMatchInput(...)` loads persisted Strategy Revisions and passes the selected adapter into both bottom and top `createRuntimeFromRevision(...)` calls.
- Runtime source-boundary tests prevent executable runtime imports and `createRuntimeFromRevision` usage in web/API source.
- Worker tests prove gameplay runtime violations complete the Match, while subprocess malformed IPC/signal failures are recorded as system failures.

Requirements covered: ISO-01 through ISO-07.

### Workshop -> Worker -> Public Replay

Status: WIRED

- Workshop client validates source, submits revisions, launches tests, polls test summaries, and renders `Open replay` links only when `hasReplay` is true.
- Service-backed E2E calls `/api/test-support/run-worker-once`, then opens `/matches/{matchId}/replay` and verifies replay UI.
- `run-worker-once` executes the worker package path in a child process rather than running Strategy code in the web request.

Requirements covered: DEBUG-03, REL-04.
