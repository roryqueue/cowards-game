# v1.26 Research Summary

**Project:** Coward's Game
**Milestone:** v1.26 Match Execution Reliability, Retry Semantics, and Failure Drills
**Domain:** Go Match orchestration reliability, runtime-service failure envelopes, retry classification, live unavailable/stopped-service drills, stale artifact detection, malformed runtime result handling, job lifecycle idempotency, and public-safe compatibility with `match-execution-app-v1`
**Researched:** 2026-05-30
**Confidence:** High for local codebase baseline and phase structure; high for no-contract-change default; medium for exact implementation details until Phase 183 inventory verifies live behavior against the current database and proof harness.

## Executive Summary

v1.26 should harden execution reliability behind v1.25's frozen `match-execution-app-v1` boundary. The app-facing contract is intentionally stable; the remaining useful work is execution-side: deterministic Go retry policy, stopped-runtime drills, stale artifact and malformed runtime result fail-closed behavior, runtime-service envelope translation, persistence/job lifecycle idempotency, and signed-in live proof that public result/replay pages remain compatible and private-data safe.

The milestone should not broaden result/replay UX, promote languages, certify a production sandbox, or migrate the active WASM/WASI ABI. JS/TS remains counted. Python, Rust, and Zig remain non-counted exhibition beta. WASI Preview 1 stdin/stdout JSON remains the active WASM/WASI execution ABI.

## Baseline Findings

- v1.25 freezes `match-execution-app-v1` in `packages/spec/src/match-execution-contract.ts` and documents the freeze in `.planning/artifacts/v1.25-interface-freeze-decision.md`.
- Go orchestration currently owns job claiming, runtime-service invocation, Match completion, Chronicle persistence handoff, MatchSet refresh, retry policy, and public evidence projection.
- `apps/go-backend/runtime_service_client.go` already distinguishes request-side validation failures from transport, timeout, malformed response, oversized response, response contract mismatch, and runtime-service system failure envelopes.
- `apps/go-backend/job_lifecycle.go` already implements bounded retry with `max_attempts`, lease tokens, retry queueing, terminal system failure, failure detail sanitization, and MatchSet refresh on exhausted failure.
- `apps/runtime-service/src/execute-match.ts` already returns schema-validated system failure envelopes for malformed requests, source hash/byte mismatches, unsupported runtime adapters, execution exceptions, invalid internal response shape, and artifact validation failures.
- v1.20 documented a retry taxonomy, but v1.26 needs executable/live drills behind the v1.25 contract, especially stopped runtime-service, malformed response, stale artifact, and public-safe result/replay proof.

## Calibration Decisions

- **Contract:** Keep `match-execution-app-v1` frozen. Do not add app-facing fields unless Phase 183 proves a strictly backward-compatible public-safe addition is necessary.
- **Retry split:** Classify by layer. Transport/envelope/system failures may be retryable while attempts remain; Strategy output failures and stale artifact paths fail closed as non-retryable public evidence.
- **Proof:** Prioritize local live drills using local Postgres, Go backend, runtime-service, internal run-once commands, result/replay pages, and private-marker scans.

## Key Uncertainties to Resolve in Phase 183

- Whether Go currently projects all terminal execution-side failures into the frozen public failure categories without relying on fixture-only metadata overrides.
- Whether runtime-service malformed HTTP envelopes and malformed Strategy/runtime outputs are visibly distinct enough in persisted job attempts and public evidence.
- Whether stale WASM/WASI artifact failures consistently become non-retryable `stale_artifact` public evidence rather than generic system failure.
- Whether stopped-runtime/unavailable-service drills can be run repeatedly without leaving duplicate running jobs, stale leases, or misleading MatchSet state.
- Whether any public-safe optional addition to `match-execution-app-v1` is genuinely needed. The default answer is no.

## Recommended Phase Structure

1. Phase 183: v1.25 contract baseline and execution-side drift inventory.
2. Phase 184: Retry and non-retry lifecycle semantics in Go orchestration.
3. Phase 185: Runtime unavailable and stopped-service live drills.
4. Phase 186: Malformed runtime result and stale artifact failure drills.
5. Phase 187: Runtime-service failure envelope translation hardening.
6. Phase 188: Persistence/job lifecycle reliability and idempotency checks.
7. Phase 189: Contract compatibility proof against `match-execution-app-v1`.
8. Phase 190: End-to-end signed-in execution reliability proof.
9. Phase 191: Audit, archive, commit, and tag.
