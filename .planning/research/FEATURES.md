# Research: Features

**Project:** Coward's Game
**Date:** 2026-05-18
**Milestone context:** v1.1 Trustworthy Simulation Beta

## Feature Thesis

v1.0 proved the author -> execute -> replay loop. v1.1 should prove trust in that loop. The player/developer outcome is not "more game modes"; it is confidence that legal Matches are generated by the engine, replay is faithful, bad Chronicles fail loudly, hostile Strategy code stays contained, and local/CI runs are repeatable.

## Table Stakes

### Replay Correctness and Fidelity

- Developer can generate replay fixtures by running legal engine scenarios instead of hand-authoring arbitrary Chronicle data.
- Developer can run canonical demo Matches covering push, fall, contraction, legal Backstab, runtime failure, and endgame.
- Developer can run fixture guard tests that reject visually useful but illegal replay beats.
- User can inspect replay states that correspond to engine-generated snapshots and event sequences.
- CI can catch board-scale, piece-position, contraction, and callout regressions with focused screenshot checks.

### Strict Chronicle Grammar and Compatibility

- Developer can validate Chronicle payload shape and semantic grammar before any replay render path accepts it.
- Invalid event order, missing required events, impossible transitions, bad snapshot references, privacy leaks, and unsupported versions fail with clear validation errors.
- Replay unavailable screens can explain Chronicle failure categories without trying to render partial nonsense.
- Chronicle compatibility behavior is documented for current, unsupported future, and unsupported legacy schema versions.
- Public replay projection remains privacy-safe by default.

### Runtime Isolation Hardening

- Developer can see exactly which runtime adapter is active and what boundary it provides.
- Prototype worker-thread execution remains tested but is not described as a final sandbox.
- Runtime adapter spike demonstrates subprocess or container-style isolation with timeout, memory/output bounds, empty env, no inherited host capabilities, and schema-validated IPC.
- Hostile Strategy tests cover forbidden globals, dynamic import attempts, worker/process access, infinite loops, memory pressure, oversized output, invalid output, and thrown exceptions.
- Worker failures distinguish strategy violation from system failure and do not run Strategy source in web/API processes.

### Doctrine Debugging UX

- User sees clearer validation messages tied to exact runtime/Strategy API constraints.
- User can start from sample Strategies that intentionally demonstrate common doctrine patterns and failure modes.
- User can open replay links directly from Workshop Match results when a replay exists.
- User can understand "why did this Soldier do nothing?" from replay context: no activation order, invalid action, blocked movement, timeout, thrown exception, STONE, FALLEN, or Match ended.
- Owner-only replay overlays surface private debug data without changing public replay projection.

### Local/CI Reliability

- Developer can start dependencies via Docker Compose or no-Docker local Postgres path with clear diagnostics.
- Developer can run migrations/seed/preflight without guessing which service is missing.
- CI covers service-backed edit -> submit -> execute -> replay beyond a smoke-only fixture path.
- Test commands separate fast unit/invariant suites from service-backed and screenshot suites.

## Differentiators for v1.1

- Chronicle grammar as a trust contract, not a loose log format.
- Engine-generated demo gallery that doubles as replay documentation and regression corpus.
- Debug UX focused on doctrine iteration rather than generic code-editor errors.
- Explicit, honest runtime boundary documentation that makes sandbox debt visible and testable.

## Anti-Features

| Feature | Reason to defer |
| --- | --- |
| Ranked ladders | Requires replay dispute handling, sandbox abuse controls, deterministic compatibility policy, stale revision policy, public/private data policy, and scoring fairness. |
| Public tournaments | Same trust risks as ladders, plus scheduling and moderation surfaces. |
| Strategy marketplace/publishing | Requires stronger privacy, abuse, ownership, and compatibility foundations. |
| Multi-language runtimes | Keep the runtime boundary replaceable, but harden JS/TS first. |
| Random/custom maps | Adds fixture and compatibility complexity while replay trust is still being sharpened. |

## Requirements Implications

Group v1.1 requirements around:

- Replay fixtures and visual fidelity
- Chronicle grammar and compatibility
- Runtime isolation and hostile-code tests
- Workshop/replay debugging UX
- Local/CI reliability

These categories map cleanly to a 4-5 phase roadmap starting after v1.0 Phase 7.
