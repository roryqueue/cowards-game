# Research Summary: v1.9 Backend and Runtime Ownership Split

**Project:** Coward's Game  
**Milestone:** v1.9 Backend and Runtime Ownership Split  
**Date:** 2026-05-22  
**Confidence:** HIGH for the ownership decision; MEDIUM-HIGH for runtime promotion-readiness details.

## Executive Recommendation

v1.9 should make one production ownership move: migrate more web read/user surfaces behind `@cowards/service` and widen strict import enforcement. Start with the public player profile read, then account session snapshot plus account revision list if the first slice lands cleanly. This is the only candidate that reduces known architecture debt while staying inside already-proven v1.8 contracts, privacy guards, local topology diagnostics, and boundary monitors.

Do not make v1.9 a backend rewrite, production sandbox promotion, or non-JS counted-play launch. Go read-model expansion remains a good follow-up, but only after a route is first canonical in `@cowards/spec`, implemented in `@cowards/service`, covered by TypeScript-generated parity fixtures, and routed explicitly with rollback. Runtime isolation should produce required live container evidence and promotion gates, not counted default execution. Python and other non-JS runtimes should remain experimental, visible, and fail-closed for counted MatchSets, ladders, and gauntlets.

The main risks are false ownership moves and privacy regressions: moving a page through a facade that still imports persistence, adding Go routes without TypeScript service ownership, treating skipped container probes as production evidence, or surfacing non-JS metadata as support parity. v1.9 requirements should make the measurable output concrete: every migrated web surface becomes strict, report-only web persistence debt decreases, public/owner-safe DTOs pass leak checks, and deferred boundaries stay explicitly blocked.

## Recommended v1.9 Scope and Sequencing

### Phase 1: Ownership Decision and Boundary Matrix

**Rationale:** v1.9 needs one deliberate move, not four partial moves. The matrix should freeze what is owned by web, TypeScript service, Go, runtime adapters, and future milestones before implementation begins.

**Delivers:**
- Route/file ownership matrix for the chosen service-backed web reads.
- Explicit non-goals for Go writes, production runtime promotion, non-JS counted play, full replay projection, and Workshop source/mutation flows.
- Baseline evidence from `pnpm boundary:imports` and `pnpm boundary:monitors`.

**Must avoid:** deciding implementation by convenience or letting a facade migration count without strict import proof.

### Phase 2: Public Player Profile Service Migration

**Rationale:** This is the best first slice. `getPublicPlayerPage` already exists in the service route registry/schema set, while the current page reaches through `competitiveServer` into persistence-backed profile data. It is public, read-only, and does not need Strategy source.

**Delivers:**
- `CowardsService` method for public player page DTOs.
- Canonical schema parsing and public leak checks before return.
- Web page rewired through the public service boundary while preserving found/not-found behavior, public Strategy cards, ladder history, and privacy copy.
- `apps/web/app/players/[handle]/page.tsx` or its chosen dependency closure added to strict migrated import enforcement.
- Report-only import baseline reduced from the current 41 known broad web offenses.

**Must avoid:** source, StrategyMemory, SoldierMemory, objective payloads, raw Awareness Grid, owner debug, stack traces, stderr, host paths, sessions, tokens, or runtime internals in public DTOs or diagnostics.

### Phase 3: Account Session Snapshot and Revision List Read

**Rationale:** This gives v1.9 a useful owner-user surface after the public read lands. It should migrate only session lookup and revision metadata/listing, not saves, source retrieval, submissions, or MatchSet creation.

**Delivers:**
- Service-owned auth session snapshot and account revision list reads, using existing route contracts where possible.
- Web cookie extraction may remain in web, but session ids and bearer tokens must never appear in service DTOs or diagnostics.
- Revision list returns metadata, hashes, status, language/runtime/counting eligibility, and compatibility data only. It must not return source.
- Preserved unauthenticated behavior and owner authorization semantics.

**Must avoid:** migrating POST routes, Strategy source retrieval, Workshop save/test flows, analytics rerun/export mutations, or using owner-specific not-found behavior as an existence oracle.

### Phase 4: Optional Public Ladder Season Read or Go Follow-Up

**Rationale:** Add only if Phases 1-3 land cleanly. Public ladder reads are plausible service migrations, but the detailed season route may need an explicit contract. Go follow-up is useful only for a route already implemented in `@cowards/service` and parity-fixture generated.

**Delivers, if ladder is chosen:**
- Explicit service route contract for detailed public ladder season read if needed.
- Public DTO leak scan and preservation of season not-found, standings, entries, MatchSet links, counted explanations, and no-permanent-ratings copy.

**Delivers, if Go follow-up is chosen:**
- Generated TypeScript-service parity fixtures for a selected public read route.
- Go GET handler, method rejection tests, privacy scans, manifest drift check, topology required-live check, and rollback flag.

**Must avoid:** Go mutations, auth/session mutation, ladder entry/schedule/status writes, Match orchestration, job claiming, migrations, Strategy execution, full replay projection, or owner analytics expansion without a stronger auth model.

### Phase 5: Runtime and Non-JS Guardrail Artifacts

**Rationale:** The runtime research should feed requirements as guardrails and future promotion criteria, not as v1.9 production promotion.

**Delivers:**
- Runtime isolation promotion-readiness criteria: required container probes, resource/fs/network evidence, image provenance, topology preflight, failure taxonomy, and redacted observability.
- Non-JS promotion criteria: deterministic language semantics, package policy, Workshop UX/docs, compatibility keys, counted eligibility, replay/export privacy, and rollback/deprecation rules.
- Monitors that continue failing on accidental Python counted eligibility or silent runtime fallback.

**Must avoid:** flipping JS/TS counted defaults, running Strategy code in web/API, treating Node `vm` as a security boundary, or adding a public language picker that implies Python support parity.

## Explicit Out-of-Scope Boundaries

- No Go mutation endpoints, auth mutation, Strategy submission, Strategy source retrieval, MatchSet creation, Match orchestration, job claiming, persistence writes, migrations, or Strategy execution.
- No full backend rewrite, split-brain persistence, cloud deployment, Kubernetes, service mesh, or production observability stack.
- No production hostile-code sandbox promotion or counted Match execution through a newly promoted runtime adapter.
- No silent runtime fallback from container/isolated adapter to worker-thread, host subprocess, JS/TS, stale fixtures, or in-process execution in required runs.
- No counted Python or other non-JS MatchSets, ladders, gauntlets, analytics comparisons, or standings by default.
- No public language picker, package dependency ecosystem, or public support copy implying non-JS parity.
- No migration of Workshop Strategy source, save, submit, validate, test MatchSet creation, analytics rerun, or export mutation flows in the first service slice.
- No migration of full replay projection or owner-debug replay data in the first ownership move.
- No rule, Chronicle, scoring, terminology, engine, or deterministic runtime semantics changes.

## Requirement Themes for v1.9

These themes should feed `.planning/REQUIREMENTS.md` only after the ownership decision is confirmed.

### Service Ownership and Import Boundaries

- Define the exact migrated files and their local dependency closure before implementation.
- Require each migrated read to be implemented in `@cowards/service`, validated by canonical `@cowards/spec` schemas, and covered by focused service tests.
- Require DTO leak assertions before return for every public and owner-safe migrated read.
- Require strict import enforcement for migrated pages/routes and removal of corresponding report-only baseline entries.
- Keep UI shape adapters in web, but keep persistence lookup and canonical DTO construction behind the service boundary.

### Go Read-Model Guardrails

- Go remains read-only and GET allowlisted unless route ownership, auth/session model, jobs, migrations, rollback, transactional semantics, and write behavior are separately specified.
- A Go route must first exist in `@cowards/spec`, be implemented by `@cowards/service`, and have TypeScript-service-generated parity fixtures.
- Go fixtures must reject source-bearing DTOs and private runtime/replay fields.
- Any web routing to Go must use explicit selection and one-switch rollback, with no silent fallback during required-boundary checks.

### Runtime Isolation Guardrails

- v1.9 can require promotion-readiness artifacts, not production counted promotion.
- Required container checks must prove filesystem denial, network denial, resource limits, output caps, timeout behavior, cgroup/runtime prerequisites, image provenance, and failure classification without fallback.
- Runtime diagnostics must be public-safe and distinguish Strategy violations from system failures.
- Strategy code remains outside web/API processes.

### Non-JS Runtime Guardrails

- Python and other non-JS runtimes remain `experimental`, `enabledForNormalPlay: false`, and `countedResultsAllowed: false`.
- `evaluateStrategyRuntimeCountedEligibility` and all exhibition, ladder, and gauntlet gates must fail closed for non-counted runtime metadata.
- Promotion criteria may be documented, but no public picker or package ecosystem should imply support parity.
- Compatibility keys must include runtime/language/package dimensions before any future counted support.

## Privacy and Deterministic-Engine Constraints

These constraints must carry directly into requirements:

- Engine logic remains pure, deterministic, serializable, and side-effect free.
- No game rules move into React components, web loaders, route handlers, Go handlers, or service DTO mappers.
- Do not use `Math.random`, `Date.now`, system time, filesystem, network, or database access inside engine logic.
- Strategy Revisions remain immutable once submitted for Match or MatchSet play.
- Strategy code is hostile and must never execute in the web/API process.
- Node `vm` is not a security boundary for untrusted Strategy code.
- Runtime boundaries validate all inputs and outputs with schemas.
- Public replay, service, Go, topology, monitor, export, analytics, and runtime outputs omit Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw Awareness Grid, owner debug, private Chronicle sections, stderr, stack traces, sessions, bearer tokens, host paths, environment values, and private runtime internals by default.
- Owner reads must authorize before resource-specific disclosure and should avoid existence oracles.
- Contract artifacts must not accidentally publish owner/internal schemas, runtime limits, source payloads, or private diagnostic strings.
- Canonical terminology remains Soldier, Match, Phase, Round, Activation, Cycle, Action, Advance, STONE, FALLEN, and Chronicle.

## Confidence Assessment

| Area | Confidence | Notes |
| --- | --- | --- |
| Service-backed web ownership | HIGH | Supported by current project state, v1.8 shipped service contracts, existing migrated slices, import monitors, and concrete direct-persistence debt clusters. |
| Go read-model expansion | HIGH for read-only guardrails; MEDIUM for immediate production routing | Go parity infrastructure exists, but additional routes should follow TypeScript service ownership and generated fixtures. Owner auth remains too primitive for broad owner reads. |
| Runtime isolation | MEDIUM-HIGH | v1.8 hostile probes and adapters are strong evaluation evidence, but production promotion lacks required live container, resource, image, deployment, and failure-taxonomy proof. |
| Non-JS promotion | HIGH | Repo semantics, monitors, and product gates consistently keep Python/non-JS experimental and non-counted. Promotion criteria are clear, but not satisfied. |
| Privacy constraints | HIGH | Multiple milestones and monitors enforce the private-data denylist; v1.9 must extend the same rules to any new DTOs, Go fixtures, diagnostics, and runtime artifacts. |

**Overall confidence:** HIGH that v1.9 should choose service-backed web read/user-surface migration as the production ownership move and leave Go/runtime/non-JS as bounded follow-up or guardrail work.
