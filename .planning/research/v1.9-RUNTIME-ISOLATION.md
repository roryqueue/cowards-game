# v1.9 Research: Production Runtime Isolation Candidate

**Project:** Coward's Game  
**Milestone context:** v1.9 Backend and Runtime Ownership Split  
**Researched:** 2026-05-22  
**Question:** Should the first v1.9 ownership move be production runtime isolation?  
**Overall confidence:** MEDIUM-HIGH

## Recommendation

**Do not make production runtime isolation the first v1.9 ownership move.** The current evidence supports a scoped v1.9 runtime-isolation evaluation/promotion-readiness milestone, but it does not yet support switching counted Match execution to a production hostile-code sandbox.

Recommendation strength:

| Decision | Recommendation | Strength | Why |
| --- | --- | --- | --- |
| Promote runtime isolation to production counted default in v1.9 | No | HIGH | v1.8 proves probes and adapter shape, not production deployment, kernel/resource enforcement, abuse handling, or repeated live container execution. |
| Make runtime isolation the first ownership move | No, unless reframed as evidence-only | MEDIUM-HIGH | The repo roadmap already identifies service-backed web read/user-surface migration as the lower-risk default first move; runtime promotion still has hard proof gaps. |
| Add v1.9 runtime work | Yes, as promotion-readiness only | MEDIUM-HIGH | v1.8 created the right ABI, candidate matrix, hostile probes, topology, and monitors; v1.9 can turn the container candidate from skipped/local-optional into repeatable evidence. |
| Promote non-JS counted play alongside isolation | No | HIGH | Non-JS semantics are product-owned but experimental; sandbox, packages, deterministic behavior, Workshop UX, and compatibility are not production-proven. |

The right v1.9 posture is: **keep counted JS/TS defaults unchanged; make containerized runtime isolation a required, repeatable, public-safe evaluation target; and define hard no-promotion gates.** If the milestone must make exactly one production ownership move, choose service-backed web read/user-surface migration first and keep runtime isolation as a parallel evidence gate.

## Current Evidence

v1.8 delivered meaningful sandbox evidence:

- `pnpm sandbox:evaluate` and `pnpm sandbox:evaluate:check` generate/check `.planning/artifacts/runtime-sandbox-evaluation.json`.
- The evaluation includes 21 hostile probes for time, randomness, filesystem, network, environment, shell/process access, dynamic code execution, malformed IPC, oversized output, stdout/stderr caps, memory/source limits, timeout, crash behavior, schema-invalid output, and thrown exceptions.
- Worker-thread baseline and host subprocess executed and passed all 21 probes.
- Container subprocess is implemented as an optional executable candidate with Docker controls, but the committed artifact records it as skipped unless `COWARDS_RUN_CONTAINER_SANDBOX=1`.
- Deno permissions, WASM/WASI, gVisor/runsc, and microVM/Firecracker remain tradeoff-only, not repo adapters.
- v1.8 fixed two real drifts: worker Strategy code could call `console`, and subprocess output caps were enforced at the IPC layer before child-harness output caps.
- v1.8 monitors compose sandbox artifact checks with contract, privacy, import-boundary, Go parity, topology, and adapter drift checks.

This is strong **evaluation evidence**, not strong **production ownership evidence**.

## Why Promotion Is Premature

### Worker Threads Are Not Enough

Node's `node:vm` is explicitly not a security boundary for untrusted code, so it remains excluded. Node worker threads provide useful resource controls for a JS engine instance and can receive isolated environment settings, but they are still a compatibility/prototype containment layer for this project. v1.8's own metadata says worker-thread execution has host filesystem and inherited network exposure at the boundary level.

### Host Subprocess Is Not Enough

The host subprocess adapter improves failure isolation and JSON IPC taxonomy, but it still inherits host filesystem and network policy unless the Strategy harness blocks access before escape. That is not a production hostile-code isolation argument.

### Container Subprocess Is the Best Current Candidate, But Not Proven

The repo's container adapter has the right shape: `--network none`, read-only root filesystem, tmpfs scratch, memory/CPU/PID limits, dropped capabilities, no-new-privileges, no shell, JSON IPC, timeout, and public-safe system-failure mapping.

The gap is that v1.8 did not require live container execution in normal verification, did not pin or scan images, did not prove behavior under Linux cgroup variants, did not evaluate gVisor/runsc, and did not document deployment/rollback semantics. Docker's own docs also make resource behavior host-dependent, especially around memory/swap/cgroups. That means "container candidate exists" is not equivalent to "production runtime ownership can move."

### Deno, WASI, gVisor, and Firecracker Are Not Shortcut Replacements

Deno's permission model is useful evidence for deny-by-default host APIs, but Deno's docs still recommend layered defenses for arbitrary untrusted code, including OS sandboxing and VM/microVM isolation. It also introduces import/package/product semantics that Coward's Game has not promoted.

Wasmtime/WASI is promising for deterministic fuel-based interruption and capability-shaped host APIs, but it would be a new runtime architecture and adapter/package model. Wasmtime's `ResourceLimiter` primarily addresses Wasm instance resources, while CPU needs fuel or epoch interruption. Fuel is deterministic but has overhead; epochs are less deterministic.

gVisor/runsc can harden container execution, but official setup requires host/container runtime installation and daemon configuration. Firecracker/microVM isolation is stronger but brings jailer, cgroup, chroot, image, kernel, cold-start, and fleet-operation work that is too large for a first v1.9 ownership move.

## Missing Evidence

| Area | Missing proof | Required before promotion |
| --- | --- | --- |
| Live container execution | Container probes are skipped by default in committed evidence. | CI/Linux path that runs all hostile probes in Docker without fallback, plus local opt-in docs. |
| Resource limits | Memory, CPU, PID, timeout, stdout/stderr, source, StrategyMemory, SoldierMemory, and objective caps are not proven under production-like load. | Matrix with normal, adversarial, and concurrent workloads; cgroup behavior documented; system failure vs Strategy violation classification stable. |
| Filesystem denial | Container has read-only root and tmpfs design, but no required live proof. | Probes that attempt read/write across `/`, `/tmp`, `/proc`, `/sys`, home, cwd, mounted paths, symlinks, and package cache locations. |
| Network denial | Container adapter requests no network, but live denial is not mandatory. | Probes for DNS, TCP, UDP, localhost, metadata IPs, IPv6, and inherited proxy env vars. |
| Image provenance | Default image is `node:24-alpine`, not pinned by digest in current adapter code. | Digest pinning, image update policy, SBOM/vulnerability review, and reproducible harness packaging. |
| Deployment topology | No production runtime service/process ownership contract exists. | Separate runtime service/queue boundary, health/readiness, concurrency caps, retry policy, rollback switch, and no web/API execution path. |
| Failure taxonomy | v1.8 improved exact result-kind matching, but production failures are broader. | Stable mapping for image pull failure, daemon unavailable, cgroup unsupported, OOM kill, seccomp/capability denial, network setup failure, timeout, malformed IPC, adapter crash, and host pressure. |
| Observability | Local artifact checks exist; production-style metrics/log redaction do not. | Public-safe metrics for counts/codes/timing/cold start/resource class without source, memory, objectives, stderr, stack traces, tokens, or host paths. |
| Local ergonomics | `COWARDS_RUN_CONTAINER_SANDBOX=1` opt-in avoids flake but also avoids evidence. | Clear `pnpm sandbox:evaluate:container`, preflight diagnostics, image pull/build docs, skip semantics, and no silent fallback. |
| Abuse model | Hostile probes are broad but not an abuse-rate model. | Queue quotas, per-user limits, match cancellation policy, noisy-neighbor tests, and operator kill/disable controls. |

## Viable v1.9 Scope If Runtime Is Chosen

If v1.9 chooses runtime isolation work, it should be an **evaluation/promotion-readiness milestone**, not a production switch:

1. **Container Evidence Required Path**
   - Add a required Linux/Docker verification lane that runs the full hostile probe matrix against `container-subprocess`.
   - Keep worker-thread as local-dev fallback and current counted default.
   - Fail if the container path silently falls back to host subprocess or worker-thread.

2. **Runtime Adapter Promotion Contract**
   - Add spec-owned promotion states such as `experimental`, `candidate`, `shadow`, `production-counted`.
   - Require resource controls, filesystem/network controls, image provenance, failure taxonomy, privacy guarantees, and deployment readiness before state advancement.
   - Keep Strategy Revisions immutable and runtime adapter metadata explicit in compatibility keys.

3. **Resource and Boundary Hardening**
   - Prove timeout, CPU, memory, PID, stdout/stderr, output payload, source byte, StrategyMemory, SoldierMemory, and objective limits under normal and adversarial load.
   - Add cgroup/version diagnostics and host-capability warnings to topology output.

4. **Isolated Runtime Service Shape**
   - Define a process/service boundary that accepts only `strategy-runtime-abi-v1.7+` envelopes and returns only schema-validated public-safe results.
   - No Strategy source execution in web/API process.
   - No direct web/API imports of runtime adapters.
   - System failures remain degraded/non-counted unless explicitly classified otherwise.

5. **Shadow-Only Execution**
   - Run optional non-counted shadow MatchSets or gauntlets through the container adapter and compare deterministic outputs against current counted JS/TS behavior.
   - Do not publish private runtime data. Do not make shadow evidence counted standings.

## Adapter Requirements

The promoted adapter contract must require:

- Stable adapter id/version and ABI version.
- Explicit readiness state and counted eligibility.
- No fallback from isolated adapter to less isolated adapter in required runs.
- Strict input/output schemas at the runtime boundary.
- Deterministic public result normalization through canonical Strategy/Soldier result schemas.
- Separate classification for Strategy runtime violations and system failures.
- Stable public error codes with private details retained only in authorized/operator channels.
- Capability metadata for filesystem, network, environment, shell/process, package access, clock/randomness, memory, CPU, PID, stdout/stderr, source, and output limits.
- Compatibility metadata recorded in Match/MatchSet evidence without exposing Strategy source or runtime internals.

## Resource, Filesystem, and Network Requirements

Minimum promotion bar:

- CPU: wall-clock timeout plus container CPU cap; define whether CPU throttling leads to timeout, system failure, or retry.
- Memory: hard memory cap, swap behavior documented, OOM mapped to system failure unless the Strategy exceeded a declared logical memory limit.
- PID/process: PID cap and shell/process-spawn denial.
- Filesystem: read-only root, no host mounts except controlled harness material, tmpfs scratch with `noexec,nosuid` where supported, write attempts classified safely.
- Network: no outbound network, no DNS, no localhost/metadata access, no inherited proxy/token env data.
- Environment: minimal allowlist, no secrets, no user/session/runtime-private env.
- Output: stdout/stderr byte caps, Strategy output byte caps, schema validation, public-safe messages.
- Source/package: source byte cap, no ad hoc package install, image/package provenance policy before any dependency support.

## Deployment Requirements

Runtime isolation promotion needs deployment evidence, not only adapter code:

- Dedicated runtime service or worker process ownership separated from web/API.
- Required preflight for Docker/container runtime, image digest, cgroup support, network disablement, and tmpfs/read-only-root support.
- Health/readiness endpoint that reports adapter id/version, ABI version, limits, isolation mode, and readiness without secrets.
- Queue/concurrency controls and backpressure.
- Rollback flag that disables isolated runtime promotion without falling back inside the same counted Match.
- CI lane that runs live container hostile probes.
- Local lane that is explicit when Docker is absent.
- Observability that redacts source, StrategyMemory, SoldierMemory, objective payloads, raw Awareness Grid, stderr, stack traces, sessions, tokens, host paths, and private runtime internals by default.

## Failure Taxonomy Requirements

Production runtime isolation must classify these distinctly:

| Failure | Classification | Public behavior |
| --- | --- | --- |
| Invalid Strategy output | Strategy runtime violation | Counted against Strategy according to existing policy. |
| Forbidden capability attempt caught by validator/harness | Strategy runtime violation | Public code only; no source excerpt or stack trace. |
| Timeout from Strategy loop | Strategy runtime violation if within healthy adapter | Existing timeout code; no internals. |
| Container image missing/pull failure | System failure | Degraded/non-counted, retryable/operator-visible. |
| Docker daemon/runtime unavailable | System failure | Degraded/non-counted; topology/preflight should catch. |
| cgroup/resource feature unsupported | System failure/preflight failure | Do not start counted runtime. |
| OOM kill | Needs explicit policy | Prefer system failure until logical memory attribution is proven. |
| Seccomp/capability denial | Strategy violation only if caused by Strategy capability attempt; otherwise system failure | Public code must not reveal host details. |
| Malformed IPC response | System failure unless produced by validated Strategy output path | Existing taxonomy should remain. |
| Adapter crash | System failure | Degraded/non-counted, no stack/stderr public leak. |

## Local Ergonomics

v1.9 should avoid making every developer require Docker for basic work, but it must stop treating skipped container execution as promotion evidence.

Recommended local split:

- `pnpm sandbox:evaluate` keeps current always-available worker/host evidence.
- `pnpm sandbox:evaluate:container` requires Docker and runs all hostile probes against container subprocess.
- `pnpm sandbox:evaluate:check` verifies deterministic artifacts.
- `pnpm topology:check --require-runtime-container` fails if container runtime prerequisites are missing.
- Diagnostics show adapter id, ABI version, image digest, Docker availability, cgroup/resource support, network mode, filesystem mode, and skipped/preflight reasons with privacy redaction.

## Privacy Risks

Runtime isolation work creates new leak channels:

- Strategy source in diagnostics, probe reports, IPC payload logs, image build output, crash dumps, or stderr.
- StrategyMemory, SoldierMemory, objectives, raw Awareness Grid, runtime internals, stack traces, host paths, tokens, sessions, or environment values in public DTOs, topology output, monitor output, or replay evidence.
- Container/runtime error messages exposing image names, host paths, daemon socket paths, cgroup paths, kernel details, or internal command lines.
- Shadow-run comparisons accidentally publishing private runtime data as evidence.

Mitigation: every runtime artifact and service DTO must pass the existing public-private denylist discipline and add container/runtime-specific redaction for image, host, cgroup, daemon, and stderr details.

## Explicit No-Promotion Criteria

Do not promote production runtime isolation in v1.9 if any of these are true:

- Container hostile probes are skipped in the required verification lane.
- Required runtime can silently fall back to worker-thread or host subprocess.
- Filesystem or network denial lacks live proof.
- Docker image is unpinned or image provenance/update policy is undefined.
- cgroup/resource controls are unavailable or unreported in required deployment topology.
- Failure taxonomy cannot distinguish Strategy violation from system failure for OOM, daemon failure, image failure, malformed IPC, timeout, and adapter crash.
- Public or diagnostic outputs expose Strategy source, StrategyMemory, SoldierMemory, objectives, stderr, stack traces, host paths, tokens, sessions, or private runtime internals.
- Local developer path becomes too brittle to run normal non-runtime work.
- Non-JS counted eligibility is bundled into the same change.
- Match orchestration, backend rewrite, or Go write ownership is bundled into the same change.

## Counted-Play Promotion

Counted-play promotion should remain premature until:

- The isolated JS/TS container path is proven under required CI and local preflight.
- Current JS/TS counted behavior is preserved or compatibility changes are explicitly versioned.
- Shadow MatchSets prove deterministic equivalence for representative Strategies and hostile failures.
- Runtime failure taxonomy has no ambiguous public scoring behavior.
- Privacy monitors cover runtime/container diagnostics and public replay/MatchSet outputs.

Non-JS counted play is further away. Python and other non-JS runtimes need package policy, reproducible lockfiles, deterministic API semantics, Workshop validation, documentation, compatibility keys, and the same isolation proof. v1.9 should not promote non-JS counted MatchSets, ladders, or gauntlets.

## Bottom Line

The first v1.9 production ownership move should not be runtime isolation. The first runtime-specific v1.9 move, if chosen, should be **container isolation promotion evidence**: required live container probes, resource/fs/network proof, deployment preflight, strict failure taxonomy, local ergonomics, and privacy-safe diagnostics. Counted-play promotion should stay blocked until those gates pass.
