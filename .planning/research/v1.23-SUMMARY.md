# v1.23 Research Summary

**Project:** Coward's Game
**Milestone:** v1.23 WASM/WASI Rust/Zig Exhibition Beta and ABI Readiness
**Domain:** WASM/WASI multi-compiler Strategy execution, Rust/Zig exhibition beta gates, ABI evolution, Zig ergonomics, runtime hardening, signed-in proof evidence
**Researched:** 2026-05-25
**Confidence:** High for continuing Preview 1 JSON execution through Wasmtime; high for Rust WASIp1 continuity; medium for Zig beta ergonomics until helper imports are audited; medium for direct-export proof; medium-low for component model/WIT promotion this milestone because cross-language tooling and product integration need evidence.

## Executive Summary

v1.23 should attempt a narrow product promotion from non-counted exhibition alpha to non-counted exhibition beta for Rust and/or Zig, not a runtime-platform or sandbox promotion. The milestone should keep JS/TS as the counted Strategy path, Python as non-counted exhibition beta, and Go as owner of orchestration, persistence-facing backend behavior, Match lifecycle, scoring, public evidence, retry policy, and promotion decisions.

The most stable execution path remains WASI Preview 1 stdin/stdout JSON through runtime-service / Runtime Broker / Wasmtime. Rust's `wasm32-wasip1` target is explicitly a WASI Preview 1 target, and Rust renamed the older `wasm32-wasi` target to avoid confusion with Preview 2. Zig still documents `wasm32-wasi` compilation, but v1.22 showed Zig std-backed output can import capabilities outside the allowed lane. Zig beta readiness therefore depends on proving safe helpers or starter layers do not widen imports beyond the current audited boundary.

Direct exports and the component model/WIT are worth a dedicated proof spike in v1.23, but they should remain evidence unless they demonstrate deterministic memory ownership, schema validation, caps, Rust/Zig parity, privacy behavior, and rollback clarity. Component model/WIT is strategically attractive because WIT defines typed worlds and Wasmtime supports components, but using it as a hidden runtime path would be premature.

## Stack Findings

- Rust `wasm32-wasip1` remains the correct Rust target for WASI Preview 1 execution. The target assumes WASIp1 syscalls and exists separately from future Preview 2 targets.
- Zig's documented WASI target remains `wasm32-wasi`; v1.22 local proof used Zig `0.16.0` with no-std code and only `fd_read` / `fd_write`.
- Wasmtime remains suitable as the embedding/runtime lane because it supports WebAssembly, WASI, the component model, configurable CPU/memory controls, fuel/interruption, and component APIs.
- WASI 0.2 / Preview 2 and the component model are the direction of travel, but the project should treat them as an ABI migration candidate, not as a reason to abandon the proven Preview 1 JSON path inside a beta-label milestone.
- Node `node:wasi` remains unsuitable as a hostile-code sandbox boundary.

## Feature Findings

- Beta criteria should be product-facing and evidence-backed: signed-in save, immutable artifact metadata, runtime-service execution, public-safe labels, result/replay visibility, no silent fallback, and JS/TS counted regression safety.
- Promotion outcomes should be split-capable because Rust has more mature ergonomics than Zig while Zig currently depends on a deliberately tiny no-std path.
- Zig ergonomics must improve without importing forbidden filesystem, network, clock/random, path, poll, environment, or package-like capabilities.
- Signed-in proof must cover one JS/TS Strategy Revision, one Rust WASM/WASI Strategy Revision, one Zig WASM/WASI Strategy Revision, JS/TS-vs-Rust, Rust-vs-Rust, Rust-vs-Zig, and Zig-vs-Zig non-counted exhibitions, plus MatchSet result and replay page checks.
- Public evidence must expose safe language/runtime/artifact status without revealing Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw diagnostics, host paths, env values, tokens, DB DSNs, package paths, or private runtime internals.

## Architecture Findings

- Runtime-service should continue to own compile/validation/execution of Rust and Zig artifacts through registered runtime implementations and schema-validated ABI envelopes.
- Go should continue to own account persistence-facing behavior, MatchSet creation, orchestration, retry/no-retry policy, scoring, public result evidence, replay metadata, and final promotion decisions.
- The engine must remain pure, deterministic, serializable, and side-effect free; no Rust/Zig rules logic should move into React, Go, or web/API code.
- ABI proof artifacts should compare three lanes:
  - Preview 1 stdin/stdout JSON: current executable path, capped streams, schema validation, compiler-agnostic.
  - Direct exports: possible overhead reduction, but requires allocation, string/buffer, memory ownership, and per-language glue proof.
  - Component model/WIT: typed long-term contract candidate, but requires language/toolchain parity proof and versioned compatibility policy.
- Artifact compatibility evidence should record target triple, WASI profile, ABI envelope, toolchain version, artifact hash, byte count, validation status, retention policy, rollback rule, and stale/missing artifact behavior.

## Watch Out For

- Do not let "beta" mean counted, ranked, ladder, gauntlet, broad production multi-language, or production sandbox promotion.
- Do not run Strategy code in web/API/Go.
- Do not let Rust, Zig, Python, or TypeScript become backend owners.
- Do not silently fall back to JS/TS, source execution, stale artifacts, or alternate runtimes when Rust/Zig artifact execution fails.
- Do not expose Zig helpers unless import/capability evidence stays inside the accepted WASI boundary.
- Do not present direct exports or component model/WIT as active execution paths unless the milestone explicitly proves and approves them.
- Do not let public MatchSet/replay evidence leak private Strategy data, private runtime internals, raw diagnostics, host paths, environment, tokens, database URLs, or objective payloads.
