# v1.9 Research: Non-JS Runtime Promotion Decision

**Project:** Coward's Game  
**Milestone context:** v1.9 Backend and Runtime Ownership Split  
**Researched:** 2026-05-22  
**Question:** Should the first v1.9 ownership move be non-JS runtime promotion criteria, or should non-JS remain experimental while another boundary moves first?  
**Recommendation strength:** STRONG: keep Python and other non-JS runtimes experimental and non-counted in v1.9; make another ownership boundary move first.

## Recommendation

The first v1.9 ownership move should **not** be non-JS runtime promotion. Non-JS promotion criteria should be written down and kept visible, but Python and any other non-JS runtime should remain experimental, non-counted, and fail-closed for MatchSet, ladder, and gauntlet eligibility until a later milestone proves runtime isolation, deterministic execution, package policy, UX, documentation, compatibility, and monitor coverage together.

The best first v1.9 ownership move is the roadmap default: **migrate more web read/user surfaces behind `@cowards/service` and widen strict import enforcement**. That move reduces direct persistence/runtime coupling without changing Strategy execution, counted-play fairness, or public language promises. It also gives future runtime and backend moves a cleaner boundary to attach to.

Promotion criteria can be defined in v1.9 as a design contract, but satisfying them should not be the main ownership move unless v1.9 is deliberately reshaped into a production runtime milestone. Current evidence supports "criteria definition only," not "promotion."

## Current Product and Runtime Semantics

Current state is intentionally conservative:

| Area | Current behavior | Implication |
| --- | --- | --- |
| Supported counted authoring | JS/TS Strategy Revisions through `runtime-js` metadata. | Counted play is already stable and should remain default. |
| Python runtime | `runtime-python-subprocess-experimental`, version `0.1.0-experimental`, readiness `experimental`, `enabledForNormalPlay: false`, `countedResultsAllowed: false`. | Python is an ABI proof, not production support. |
| Product semantics | Account, exhibition, public Strategy card, and validation surfaces show compact runtime/readiness/counted labels. | The product can explain non-JS status without implying parity. |
| Counted gates | Exhibition and ladder gates use spec-owned counted eligibility and reject Python with stable public copy. | Counted play already fails closed for known non-JS runtime metadata. |
| Sandbox evidence | v1.8 sandbox harness is evaluation-only and explicitly says no candidate is promoted. | Runtime isolation remains evidence, not certification. |
| Drift monitors | `pnpm boundary:monitors` fails if Python becomes enabled/countable or runtime semantics drift. | Current guardrails are designed to prevent accidental promotion. |

Important implementation details:

- `packages/spec/src/runtime.ts` is the source of truth for languages, adapters, readiness, package policy, validation issue codes, counted eligibility, and user-facing runtime semantics.
- `runtime-python-subprocess-experimental` uses local Python subprocess execution with empty environment, simple JSON ABI envelopes, and no counted-play eligibility.
- The Python host executes user source with a constrained builtins dictionary, but this is not equivalent to production hostile-code isolation.
- Existing counted JS/TS runtime adapters include worker-thread, host subprocess, and container-subprocess metadata, but even the container path is still only a production candidate.
- v1.8 docs and monitors intentionally separate "production-candidate sandbox" from "production-supported runtime."

## Counted-Play Risks

Promoting non-JS counted play first would combine too many unresolved risk classes:

| Risk | Why it matters for counted play | Current status |
| --- | --- | --- |
| Isolation gap | Hostile Strategy code must not access filesystem, network, env, shell, clocks, randomness, host process memory, or private diagnostics. | Not proven for Python; JS/TS sandbox is still not final production isolation. |
| Determinism gap | Same Strategy Revision must produce identical results across machines, processes, versions, and reruns. | ABI exists, but Python version, runtime flags, locale, hash behavior, and package state are not certified. |
| Package/supply-chain gap | Dependencies affect behavior, attack surface, reproducibility, and fairness. | Package metadata is currently unsupported for counted play. |
| Failure taxonomy drift | Strategy-accountable failures and system failures must be separated for standings, evidence bands, retries, and public copy. | Python mirrors ABI shape only; it lacks the same hostile matrix depth as JS/TS adapters. |
| Product promise gap | Public language support implies examples, docs, validation messages, support burden, and stable semantics. | Product only has compact experimental labels and docs references, not a real authoring experience. |
| Evidence comparability | Gauntlets, ladders, and MatchSets compare results; runtime ABI/adapter/package/limits changes can make evidence non-equivalent. | Compatibility keys include runtime dimensions, but non-JS comparison policy is not yet product-proven. |
| Privacy leakage | Runtime failures must not expose source, memory, objectives, stderr, stack traces, host paths, tokens, or internals. | General privacy monitors exist, but Python-specific public failure surfaces are not broad enough. |

Because counted standings are player-trust surfaces, non-JS must fail closed unless every required promotion gate is satisfied. A partial promotion would make MatchSet and ladder evidence harder to explain and could force a rollback of public results.

## Required Promotion Criteria

A future promotion milestone should require all of the following before any non-JS runtime becomes counted-play eligible.

### 1. Runtime and Sandbox Criteria

- A production sandbox candidate is selected for the promoted language and is no longer evaluation-only.
- Filesystem, network, env, shell/process spawning, host path access, clocks, randomness, dynamic code loading, worker spawning, WebAssembly/native extension behavior, stdout/stderr, source size, output size, memory size, CPU, wall timeout, process count, and crash behavior are covered by hostile probes.
- The sandbox produces deterministic public-safe outcomes under repeated local and CI runs.
- System failures are retriable/degraded according to existing MatchSet/evidence rules and never become player-accountable runtime violations.
- Runtime adapter metadata and executable behavior agree on readiness, limits, isolation boundary, ABI version, package policy, and counted eligibility.
- The promoted adapter is not reachable from web/API processes and cannot execute Strategy code in-process.

### 2. Language Semantics Criteria

- The language API is documented in canonical Coward's Game terms without changing Soldier, Match, Phase, Round, Activation, Cycle, Action, Advance, STONE, FALLEN, or Chronicle terminology.
- Inputs and outputs are schema-validated against the same ABI/game schemas as JS/TS.
- Strategy source is immutable once submitted for Match or MatchSet play.
- Returned StrategyMemory, SoldierMemory, objectives, and output payloads obey the same byte limits and privacy boundaries.
- Runtime compatibility keys include ABI, language version, adapter version, package mode, source hash, spec version, engine version, required capabilities, and limits.
- Promotion defines whether the language version is fixed, image-pinned, semver-ranged, or per-adapter-versioned.

### 3. Package and Dependency Criteria

- Counted play either continues to require single-file/no-package source or defines a reproducible package policy.
- If packages are allowed, manifest and lockfile hashes are mandatory, dependency resolution is deterministic, native modules are denied or separately sandboxed, and supply-chain allow/deny policy is explicit.
- Package install/build steps do not run during Match execution.
- Package metadata appears in compatibility keys, replay provenance, service DTOs where appropriate, and owner-only diagnostics without leaking private source or secrets.

### 4. Product and UX Criteria

- Workshop authoring supports the promoted language intentionally, including templates/examples, validation, save/test flow, failure messages, source limits, and runtime labels.
- Public language/support surfaces distinguish production-supported languages from experimental languages.
- Any language picker appears only after at least one non-JS runtime is production-supported; before that, no public picker should imply support parity.
- Exhibition, ladder, and gauntlet entry UI explains counted eligibility and disables non-counted revisions with stable copy.
- Public Strategy cards show language, adapter, readiness, package policy, and counted status without exposing source or runtime internals.
- Docs include examples, deterministic restrictions, forbidden capabilities, package policy, failure taxonomy, counted eligibility, and rollback/deprecation policy.

### 5. Evidence and Fairness Criteria

- Counted MatchSet creation, ladder entry, gauntlet creation/rerun, analytics comparison, exports, public profiles, and public Strategy cards all agree on eligibility.
- Saved gauntlet comparisons refuse to compare across behavior-significant runtime differences unless policy explicitly allows it.
- Public standings and evidence bands explain system-failed, degraded, non-counted, and invalid outcomes consistently.
- Replays and exports omit Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw Awareness Grid, stderr, stack traces, host paths, tokens, and private runtime internals by default.
- Rollback policy defines what happens if a promoted runtime is later found unsafe or nondeterministic.

### 6. Validation and Monitor Criteria

Promotion should require green checks for:

- `pnpm boundary:monitors`
- `pnpm sandbox:evaluate:check` or its promoted-runtime successor with promotion-specific gates
- runtime adapter executable/spec drift checks
- Python/non-JS hostile capability matrix
- counted exhibition rejection/acceptance tests
- ladder eligibility tests
- gauntlet eligibility and compatibility tests
- public DTO/privacy leak tests
- replay reconstruction and export privacy tests for promoted-language evidence
- local topology diagnostics including promoted runtime process readiness
- CI proof that missing runtime dependencies fail loudly instead of falling back to JS/TS or stale fixtures

The monitors should invert only when promotion is deliberate: today they should fail if Python becomes counted; in a future promotion milestone they should fail unless Python is counted only through the promoted adapter and all criteria above pass.

## Product and Documentation Prerequisites

Before non-JS promotion, product surfaces need more than labels:

- A real language-specific Workshop path, not only metadata visibility.
- Language-specific starter examples and minimal deterministic templates.
- Stable public docs for runtime/languages, runtime/adapters, runtime/package-policy, runtime/capabilities, runtime/limits, runtime/counting, and examples for the promoted language.
- Clear copy that "experimental" means not eligible for counted MatchSets, ladders, or gauntlets.
- A public support matrix that lists JS/TS as production-supported and non-JS runtimes as experimental until promotion.
- Owner-only diagnostics for promoted-language runtime failures that remain absent from public replay/service/export DTOs by default.

Until these exist, adding a normal public language picker or marketing-style language support copy would be misleading.

## Runtime and Sandbox Dependencies

Promotion depends on work that is broader than "turn on Python":

- Production hostile-code isolation must be selected and proven first, likely starting from container subprocess or another hardened process boundary.
- Runtime process ownership must be explicit: which service starts it, how it is configured, how it is observed, how timeouts/caps are enforced, and how crashes are classified.
- Local topology must include the promoted runtime as a required boundary when counted play depends on it.
- Adapter configuration must not silently fall back to worker-thread, JS/TS, stale fixtures, or in-process execution.
- The Python host needs a security review; `exec` with limited builtins is useful for an ABI spike but should not be treated as a sandbox.
- Python versioning and image/runtime pinning need to be reproducible across developer, CI, and production-like environments.
- Package policy must be decided before supporting more than self-contained source.

## Recommended v1.9 Shape

Recommended ordering:

1. **Service-backed web/read ownership move.** Move more read/user surfaces behind `@cowards/service` and widen strict import enforcement. This is the safest first ownership move because it reduces coupling without changing runtime trust or counted evidence.
2. **Promotion criteria design contract.** Add a v1.9 artifact/spec section that freezes the criteria above and keeps Python experimental. This prevents accidental support creep while making the future promotion path concrete.
3. **Runtime isolation follow-up, not promotion.** If v1.9 has runtime capacity, deepen sandbox evidence for the selected production candidate. Do not make non-JS counted until the sandbox itself is production-owned.
4. **Future non-JS promotion milestone.** Promote one language only after runtime isolation, package policy, UX, docs, compatibility, and monitor criteria are all satisfied.

## Failure Modes to Watch

| Failure mode | Warning sign | Mitigation |
| --- | --- | --- |
| Criteria becomes implicit promotion | Runtime registry flips `enabledForNormalPlay` or `countedResultsAllowed` for Python. | Keep monitors failing on Python counted eligibility in v1.9. |
| Public surfaces imply parity | Language picker, docs, cards, or examples present Python next to JS/TS without experimental copy. | Use explicit support matrix and keep picker out of normal Workshop until production support exists. |
| Counted gate bypass | Exhibition, ladder, gauntlet, API, or direct persistence path accepts non-JS metadata. | Centralize all counted checks on `evaluateStrategyRuntimeCountedEligibility`; add gauntlet-specific tests if missing. |
| Sandbox evidence overclaimed | "Production-candidate" is read as production-ready. | Separate readiness labels from counted eligibility; require promotion-specific checks. |
| Runtime fallback hides dependency failure | Missing Python/container runtime falls back to JS/TS, worker-thread, or fixture behavior. | Required topology checks must fail loudly. |
| Failure taxonomy collapses | Python crash/timeout becomes player loss or public stack trace. | Require system-failure versus runtime-violation parity tests and privacy scans. |
| Evidence comparisons become unfair | Analytics compares JS/TS and Python results without runtime compatibility policy. | Compatibility keys and gauntlet comparison gates must include runtime/package dimensions. |

## Explicit Non-Goals For v1.9

- Do not enable Python or any other non-JS runtime for counted MatchSets, ladders, or gauntlets by default.
- Do not add a normal public language picker implying support parity.
- Do not support package installation or dependency manifests for counted non-JS Strategies.
- Do not promote the Python subprocess host as a production hostile-code sandbox.
- Do not execute Strategy code in the web/API process.
- Do not use Node `vm` as a security boundary for untrusted Strategy code.
- Do not rewrite backend ownership, runtime isolation, and non-JS counted play in the same milestone.
- Do not expose Strategy source, StrategyMemory, SoldierMemory, objective payloads, stderr, stack traces, host paths, tokens, or private runtime internals in public replay/service/export/topology outputs.

## Validation Checks For v1.9 Requirements

Recommended v1.9 requirement checks:

- Python registry record remains `experimental`, `enabledForNormalPlay: false`, and `countedResultsAllowed: false`.
- `evaluateStrategyRuntimeCountedEligibility` rejects Python with `NON_COUNTED_RUNTIME`.
- Exhibition and ladder tests continue rejecting Python/non-counted runtime metadata.
- Add or confirm gauntlet counted eligibility checks fail closed for Python/non-JS runtime metadata.
- Account, public Strategy card, and exhibition UI continue showing "Experimental" or "Not counted" semantics where non-JS appears.
- No public Workshop language picker is introduced unless promotion criteria are satisfied.
- `pnpm boundary:monitors` remains green and continues to fail on accidental Python promotion.
- `pnpm sandbox:evaluate:check` remains explicit that no candidate is promoted unless a future promotion-specific command replaces it.
- Local topology diagnostics do not report Python as a required production runtime boundary.
- Privacy scans cover any new non-JS docs/examples/service DTOs that mention runtime state.
