# v1.9 Research: Service-Backed Web Ownership Move

**Project:** Coward's Game  
**Milestone candidate:** v1.9 Backend and Runtime Ownership Split  
**Researched:** 2026-05-22  
**Question:** Should the first v1.9 ownership move be service-backed web read/user-surface migration and stricter import enforcement?  
**Recommendation strength:** HIGH, with a narrow first slice.

## Recommendation

Yes. The first v1.9 ownership move should be to migrate more web read/user surfaces behind `@cowards/service` and widen strict import enforcement.

This is the right first ownership move because it is the only candidate that directly reduces known architectural debt while staying inside already-proven v1.8 contracts, tests, and monitors. It does not require Go to own persistence, does not promote a production hostile-code sandbox, and does not imply counted non-JS Strategy support. It also creates a cleaner future decision point: once web callers stop reaching into persistence for reads, Go read-model expansion or runtime process isolation can be evaluated against a smaller and sharper boundary.

Do not make v1.9 a backend rewrite. Keep TypeScript persistence, writes, auth mutation, MatchSet creation, jobs, migrations, orchestration, Strategy source handling, and Strategy execution in their current ownership lanes unless a later phase explicitly specs their route ownership, authorization, rollback, and parity model.

## Current Project Evidence

- `.planning/PROJECT.md` and `.planning/MILESTONES.md` already frame v1.9 as one deliberate ownership move, with the strongest default being service-backed web read/user migration before Go or runtime promotion.
- v1.8 shipped `@cowards/service`, generated `service-api-v1.8` OpenAPI artifacts, Go parity fixtures, topology checks, and composed boundary monitors.
- `pnpm boundary:imports` currently passes with `strict_offenses=0 report_only_offenses=41`.
- `pnpm boundary:monitors` currently passes and reports `41 known broad web offenses baseline-gated`.
- The strict migrated slice already includes service health, public MatchSet summary, public replay metadata, public Strategy page, and `apps/web/lib/public-service-boundary.ts`.
- Current `@cowards/service` methods cover health, public MatchSet summary, public replay metadata, public Strategy page, and owner analytics run summary.
- Current `apps/web` direct debt clusters around:
  - `apps/web/app/competitive/server.ts` for auth/session, account revisions, exhibitions, ladder, governance, public player profile, starter/advanced lookup.
  - `apps/web/app/workshop/server.ts` for Workshop snapshots, source retrieval, submissions, test MatchSets, analytics profile/rerun/export flows.
  - `apps/web/app/matches/server.ts` for full replay Chronicle loading and owner-debug authorization.
  - page/route imports of `competitive/server` and `workshop/server`.
- `packages/spec/src/service.ts` already defines route contracts for several non-implemented service methods, including auth session, strategy revision list/source, public player page, ladder season list, analytics routes, and MatchSet creation. That means v1.9 can extend implementation without inventing a new contract system.

## Likely Target Slices

### Slice 1: Public Player Profile Read

**Recommendation:** Do first.

**Why:** `getPublicPlayerPage` already exists in the service route registry and schema set, while `apps/web/app/players/[handle]/page.tsx` currently imports `competitiveServer`, which reaches directly into `@cowards/persistence/profiles`. This is a public read, has no Strategy source payload, and matches the v1.8 public page pattern used for Strategy cards.

**Likely files:**
- `packages/service/src/index.ts`
- `packages/service/src/service.test.ts`
- `apps/web/lib/public-service-boundary.ts`
- `apps/web/app/players/[handle]/page.tsx`
- `scripts/check-service-boundary-imports.ts`
- `scripts/check-boundary-monitors.ts`

**Proof required:**
- Public player DTO schema parse.
- `assertPublicServiceDtoLeakSafe` before return.
- Existing page behavior preserved for found/not-found states.
- Import monitor makes `apps/web/app/players/[handle]/page.tsx` strict and removes its known baseline offense.

### Slice 2: Account Session Snapshot and Account Revision List

**Recommendation:** Do second, but keep source retrieval and saves out of this slice.

**Why:** This is the most useful user-surface migration after public reads. The account page currently calls `getCurrentCompetitiveUser()` and `competitiveServer.listAccountRevisions(user)`, both backed by persistence imports in `competitive/server.ts`. Service contracts already exist for `authSession` and `listStrategyRevisions`.

**Boundary shape:** web may still own cookie extraction and response-cookie setting, but session lookup and revision-list read should go through `@cowards/service`.

**Privacy risk:** owner surfaces may legitimately return owner-scoped data, but must still omit sessions, tokens, Strategy source, memory, objective payloads, stack traces, stderr, host paths, and runtime internals by default.

**Proof required:**
- Session token never appears in DTOs or diagnostics.
- Revision list returns metadata/hash/status/runtime eligibility only, not source.
- 401/unauthenticated behavior preserved for routes and account page.
- Import baseline shrinks for account page/session GET/list revision reads, without migrating POST mutation routes.

### Slice 3: Public Ladder Season Read

**Recommendation:** Do after Slice 1 or 2 if scope allows.

**Why:** `buildTrialLadderSeasonDto` is a read surface and the page is public, but the current service registry only has `listLadderSeasons`, not a detailed `getPublicLadderSeason` route. This is still a good read-model candidate because the DTO is already public and v1.3/v1.8 privacy rules are mature.

**Risk:** ladder page combines public standings with current-user affordance text. Keep current-user lookup separate from public season service data.

**Proof required:**
- Add explicit route contract if detailed season read is migrated.
- Public ladder DTO leak scan.
- Preserve season not-found, standings, entries, MatchSet links, counted explanations, and no permanent ratings copy.
- Do not migrate ladder entry/schedule/status mutations.

### Slice 4: Workshop Analytics Read-Only Narrowing

**Recommendation:** Defer unless v1.9 needs a third slice.

**Why:** Workshop surfaces are high-value but bundled with source retrieval, validation, submissions, test MatchSet creation, local static fallback, demo analytics, and owner exports. This is too broad for the first ownership move unless split aggressively.

**Acceptable narrow target:** owner analytics run summary or profile list only, using existing service DTO concepts and owner checks.

**Avoid initially:** Workshop initial data, source endpoints, submit/test routes, analytics rerun creation, and exports.

### Slice 5: Full Replay Data

**Recommendation:** Defer.

**Why:** v1.8 already migrated replay metadata. Full replay loading crosses Chronicle storage, public projection, owner-debug authorization, fixture replay support, and private Chronicle sections. It is important but too privacy-sensitive for the first v1.9 ownership move.

## Prerequisites

- Freeze the current import baseline before edits: `pnpm boundary:imports` should show zero strict offenses and the known report-only count.
- Treat `@cowards/spec` service routes and DTO schemas as canonical. Do not add ad hoc web DTOs for migrated service reads.
- Decide the exact v1.9 migrated file list up front. Each migrated page/route must be added to `strictMigratedFiles`.
- For owner reads, define how the web adapter passes viewer identity. Cookie parsing may remain in web, but service DTOs and diagnostics must never include session ids or bearer tokens.
- Expand service tests before page rewiring. Each service method should parse canonical schemas and run privacy checks before returning.
- Update monitor baselines by removing migrated offenses rather than only adding exceptions.

## Implementation Sequence

1. **Baseline and slice declaration**
   - Record current `pnpm boundary:imports` output.
   - Choose the named migrated files for v1.9, starting with public player profile.
   - Add or confirm route ids in `SERVICE_API_ROUTES`.

2. **Service method first**
   - Add `CowardsService` methods for the selected read slice.
   - Inject persistence builders as optional dependencies for tests.
   - Build canonical DTO envelopes, run leak assertions, and parse Zod schemas.

3. **Web facade migration**
   - Extend `apps/web/lib/public-service-boundary.ts` or create a separate owner-read boundary facade.
   - Keep UI shape adapters in web; keep persistence lookup in service.
   - Preserve page/route public behavior and error shapes.

4. **Strict import ratchet**
   - Add migrated files to `strictMigratedFiles`.
   - Remove corresponding entries from `knownReportOnlyBoundaryOffenses`.
   - Ensure new direct imports of `@cowards/persistence`, runtime packages, worker entrypoints, migrations, or legacy server facades fail.

5. **Contract and monitor update**
   - Regenerate/check service contract artifacts only if public route artifact scope changes.
   - Keep owner/internal routes out of public OpenAPI output unless explicitly intended.
   - Extend monitor checks for new public examples and privacy assertions.

6. **Regression verification**
   - Run focused service/web tests for migrated slices.
   - Run boundary, contract, privacy, typecheck, and relevant browser/E2E checks.

## Privacy Risks

- **Owner identity leakage:** session ids, bearer tokens, usernames, or owner-only ids can leak through diagnostics if owner-read service boundaries are sloppy. DTOs should expose only intended user/profile fields.
- **Strategy source leakage:** account revision list and public player/Strategy surfaces must expose source hashes and metadata, not source text. Keep source retrieval as a separate owner-only route with explicit authorization.
- **Memory/objective leakage:** public and owner-safe read DTOs must omit StrategyMemory, SoldierMemory, objective payloads, raw Awareness Grid, owner debug, private Chronicle sections, stack traces, stderr, host paths, and runtime internals by default.
- **Existence oracles:** owner-scoped reads should authorize the viewer before returning resource-specific not-found details. v1.8 fixed this for Go analytics; follow that pattern.
- **Contract artifact leakage:** do not publish owner/internal service schemas in the public OpenAPI artifact by accident. v1.8 already had to remove private/runtime limit strings from public output.
- **Diagnostic leakage:** topology, monitor, service errors, and test output must preserve privacy-safe messages.

## Import-Boundary Risks

- **Facade loophole:** migrating pages to a web facade does not count if the facade imports persistence directly without being an explicit approved adapter. Keep adapter exceptions narrow and documented.
- **Legacy server dependency:** pages that import `competitive/server` or `workshop/server` inherit persistence debt even if their visible behavior is read-only.
- **Type-only imports:** type-only imports from persistence in app code still couple web to persistence DTO ownership. Prefer spec-owned types for migrated surfaces.
- **Local import transitivity:** strict files must include their local dependency closure, otherwise persistence can enter through a helper.
- **Baseline decay:** keeping old `knownReportOnlyBoundaryOffenses` after migration hides progress and can permit reintroduction. Remove migrated fingerprints.
- **Over-strict flip:** making all `apps/web/app` strict at once would likely block on Workshop, replay, auth mutation, and support-test routes. Ratchet by named slice.

## Explicit Non-Goals

- No Go mutation endpoints.
- No Go ownership of auth mutation, Strategy submission, Strategy source retrieval, MatchSet creation, Match orchestration, job claiming, migrations, persistence writes, or Strategy execution.
- No production hostile-code sandbox promotion.
- No counted Python or other non-JS Strategy play.
- No public language picker or package dependency ecosystem.
- No migration of full replay public projection or owner-debug replay data in the first slice.
- No migration of Workshop Strategy source, submit, validate, test MatchSet creation, analytics rerun, or export mutation flows in the first slice.
- No rule, Chronicle, scoring, terminology, engine, or deterministic runtime semantics changes.
- No broad backend rewrite, cloud deployment, service mesh, Kubernetes, or production observability stack.

## Decision

Proceed with service-backed web read/user-surface migration as the first v1.9 ownership move. Start with public player profile, then owner session/account revision list if the first slice lands cleanly. Treat strict import enforcement as the measurable output of each slice: every migrated page/route must become strict, and the broad baseline must shrink.

This gives v1.9 a concrete, low-blast-radius ownership shift that improves the architecture now and prepares the project for later Go or runtime ownership decisions without entangling them.
