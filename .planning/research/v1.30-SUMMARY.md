# v1.30 Research Summary

**Project:** Coward's Game
**Milestone:** v1.30 Match Intelligence Workbench Against Frozen Result and Replay Fixtures
**Workstream:** `.planning/workstreams/v1-30-match-intelligence-workbench/`
**Domain:** Public-safe tactical intelligence derived from frozen `match-execution-app-v1` result/replay DTOs and fixture-backed replay projections
**Researched:** 2026-05-31
**Confidence:** High that the milestone can stay app-side and fixture-backed; medium on the exact depth of board-control/action-mix analysis until Phase 201 inventories fixture replay events and public Chronicle projection density.

## Executive Summary

v1.30 should extend the v1.27 result/replay workbench into a tactical inspection layer without changing the frozen execution boundary. The strongest path is to derive a pure app-side "Match Intelligence" adapter from public result DTOs, replay metadata/evidence DTOs, and public Chronicle projection data already accepted by the replay page. The adapter should produce summaries, timeline annotations, jump targets, Soldier progression, board-control signals, terrain/stone occupancy, action mix, engagement summaries, and comparison panels only when public DTOs support those claims.

This is not an AI coach, execution-analysis service, runtime promotion, or contract redesign. Intelligence must degrade honestly for queued, running, failed, unavailable, stale, malformed, no-result, and missing-Chronicle outcomes. The product promise is "what can be concluded from public evidence", not "what the hidden Strategies intended".

## Baseline Findings

- v1.25 froze `match-execution-app-v1` across lifecycle, MatchSet/result, replay metadata/evidence, runtime evidence, failure evidence, privacy split, fixture scenarios, and fixture adapter gates.
- v1.27 added the fixture catalog, result page view model, replay workbench layout, timeline grouping, replay focus support, board realism proof, privacy scans, and owner-debug gating.
- v1.28 kept operations recovery private and did not expand public result/replay contracts. Public compatibility outcomes include complete, queued/running, degraded/unavailable runtime, timeout, malformed runtime result, stale artifact, system failure, strategy failure, quarantined private-only, interrupted MatchSet, missing Chronicle, and no-result.
- Public replay readiness already builds timeline entries from projected public Chronicle events and board states from replay reconstruction, with in-bounds board validation and canonical start checks.
- Owner/private projection data exists for debug, awareness grid, and inactivity explanation, but default public intelligence must not depend on owner-only payloads.

## Public Signals Available

- MatchSet lifecycle: state, terminality, retry disposition, failure category, result availability, replay availability, public message key.
- Match rows: entrant labels, Match status, public reason, replay link, replay availability, arena variant, Chronicle hash where public.
- Runtime/public evidence: counted JS/TS status, non-counted beta labels, Go/runtime-service ownership, active Preview 1 JSON ABI.
- Replay timeline: sequence, event type, Round, Activation, Cycle, public/owner privacy label, context, sanitized public payload.
- Replay board states: bounds, Soldier snapshots, terrain stones, outcome, sequence.
- Existing replay focus moments: backstab, contraction, no-advance cleanup, fall, decisive push, and late-cycle stabilization.

## Recommended Stack Shape

- Add a pure TypeScript intelligence derivation module in the web app or a small shared package only if reuse genuinely warrants it.
- Keep input schemas anchored to existing public DTOs and replay-ready DTOs; do not introduce execution-internal imports.
- Add fixture-backed tests for every frozen fixture scenario and replay-ready public fixture.
- Add browser proof on desktop and mobile for completed replay fixtures and representative degraded/unavailable states.
- Extend boundary monitors for intelligence-specific privacy, contract drift, fixture fallback, and ownership creep.

## Candidate Intelligence Outputs

- Match summary: public evidence quality, outcome availability, replay readiness, failure/degraded explanation, and confidence band.
- Momentum/turning points: public event spikes such as Soldier FALLEN, STONE, backstab, push, contraction, no-advance cleanup, and late-cycle stabilization.
- Timeline annotations: labeled sequence ranges and jump targets that reuse current replay focus mechanics.
- Soldier progression: public status over time, last visible position, FALLEN/STONE transitions, recent public event involvement.
- Board-control signals: active/STONE occupancy by side, center/edge presence, contraction pressure, terrain/stone congestion, and in-bounds visible board health.
- Action and engagement mix: counts of public event types and public combat/contact events where Chronicle projection includes them.
- Result comparison: per-entrant or per-Match evidence cards derived from MatchSet DTO rows and public replay availability.

## Top Uncertainties to Discuss Before Requirements

1. **How ambitious should "momentum" be?** It can be a conservative event-count trend over public sequences, or a richer tactical score using Soldier status, occupancy, contraction, and engagement events. The richer option is more useful but needs stricter copy to avoid pretending to know Strategy intent.
2. **Where should the intelligence adapter live?** Keeping it in `apps/web` minimizes surface area; a shared package could help tests and future analytics but risks making app-only intelligence feel like a contract expansion.
3. **How much replay-overlay UI is in scope?** Timeline annotations and jump targets are clearly supported. Full board overlays/filters are promising if implemented from public board states only, but they carry higher mobile/layout proof cost.
4. **What should completed-but-low-signal fixtures show?** Some fixtures may only support lifecycle intelligence, not tactical summaries. Requirements should explicitly allow "not enough public evidence" states instead of forcing invented insights.
5. **How deep should owner/test-only analysis go?** A gated deeper panel can review public derivation internals and fixture diagnostics, but it must not expose Strategy source, memories, objective payloads, raw diagnostics, or private runtime details by default.
6. **Does v1.30 need new public DTO fields?** The default answer should be no. If a gap appears, it should be recorded as future contract evolution rather than silently widening `match-execution-app-v1`.

## Recommended Phase Shape for Discussion

The user's proposed 201-210 range fits the repo history and should remain the starting point:

1. Phase 201: Baseline inventory of result/replay DTO intelligence signals.
2. Phase 202: Fixture-backed intelligence derivation adapter.
3. Phase 203: Result page tactical summary and comparison model.
4. Phase 204: Replay timeline annotation and jump-target workbench.
5. Phase 205: Soldier status, board-control, terrain/stone, and action-mix panels.
6. Phase 206: Degraded/unavailable/queued/running intelligence states.
7. Phase 207: Owner/test-only gated deeper analysis review.
8. Phase 208: Desktop/mobile visual proof across fixtures.
9. Phase 209: Boundary monitors, privacy audit, and live signed-in compatibility proof.
10. Phase 210: Audit, archive, commit, and tag.

## Watch Outs

- Do not derive public intelligence from owner-only Chronicle private sections, Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw diagnostics, host paths, env values, tokens, DB details, package paths, or private runtime internals.
- Do not infer Strategy intent or "coach" advice from hidden model/Strategy behavior.
- Do not let intelligence adapters import Go orchestration, runtime-service, persistence internals, or private debug payloads.
- Do not let fixture mode become a silent production fallback.
- Do not make unavailable, queued, running, failed, stale, malformed, no-result, or missing-Chronicle outcomes look tactically analyzed when public evidence is absent.
- Do not expand `match-execution-app-v1` unless a future explicit contract evolution proves a backward-compatible public-safe addition.
