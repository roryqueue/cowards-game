# Project Research Summary

**Project:** Coward's Game
**Milestone:** v1.19 Runtime Isolation Readiness and Exhibition Beta Trust
**Domain:** Runtime isolation readiness evidence, hostile Strategy probes, Python non-counted exhibition beta trust, broker topology, public replay/result evidence, privacy monitors
**Researched:** 2026-05-25
**Confidence:** High for repo-local baseline and approved phase direction; medium for container/gVisor local proof until those candidates are run on the target machine.

## Executive Summary

v1.19 should treat v1.18 as the floor rather than reopening the Python promotion question. Python is already non-counted exhibition beta; the next useful milestone is to make the evidence more trustworthy, the candidate comparison sharper, and the user-facing proof easier to understand. The central claim should remain honest: runtime isolation is readiness evidence unless a stronger sandbox candidate genuinely passes required live proof.

The best milestone spine is a candidate-evidence loop: define the subprocess/container/gVisor-style candidate contract, expand realistic hostile probes, run or fail-loud candidate evidence, and wire monitors so stale, skipped, silently substituted, or overclaimed evidence fails. The product-facing proof should be a signed-in exhibition flow with clearer Python labels, better validation messages, credible sample Python Strategies, private-data-safe MatchSet result evidence, and replay trust cues.

## Stack Findings

- Preserve the v1.18 topology: web frontend -> Go backend -> Strategy Execution Service / Runtime Broker -> isolated runtime implementation(s).
- Keep Go as orchestration, persistence-facing backend behavior, Match lifecycle, scoring, and public evidence owner.
- Keep hostile Strategy execution behind schema-validated runtime ABI envelopes.
- Python `-I` isolated mode and safe-path behavior are useful local hardening controls but not production hostile-code sandbox certification.
- Node child process controls can support timeout, stdio, environment, and shell/no-shell evidence, but subprocesses alone do not provide filesystem/network/kernel isolation.
- Docker/container controls can provide practical local readiness evidence: network denial, read-only root filesystem, tmpfs write surfaces, dropped capabilities, PID limits, memory/CPU limits, and fail-loud launch behavior.
- gVisor/runsc is a relevant stronger candidate because it is OCI-compatible and inserts a userspace kernel layer, but local availability, overhead, compatibility, and unsupported behavior must be measured rather than assumed.

## Feature Findings

- Table stakes are baseline/candidate contracts, hostile probe expansion, required candidate evidence, drift monitors, exhibition beta UX trust, public MatchSet/replay trust cues, signed-in end-to-end proof, and explicit promotion decisions.
- The v1.18 proof already established the account -> JS/TS revision -> Python revision -> non-counted MatchSet -> runtime -> replay path; v1.19 should make that proof more legible and harder to regress.
- User-facing labels should consistently say "non-counted exhibition beta" wherever Python can be selected or observed.
- Public proof outputs should explain language/runtime/counting status without exposing Strategy source, memory, objective payloads, stderr, stacks, host paths, package paths, tokens, DB DSNs, sessions, or private runtime internals.

## Architecture Findings

- Runtime candidate selection must remain exact-match and fail-closed through the Runtime Broker registry and ABI metadata.
- Required candidate lanes should fail when a candidate is unavailable or skipped; optional evidence may be recorded, but it must not be silently substituted for promotion evidence.
- Validation and execution import surfaces must remain separate so validation paths do not pull runtime execution adapters into Next/web evaluation.
- Topology checks should distinguish fixture-mode parity from live signed-in account proof data to avoid false confidence or false failures.
- Monitors should extend the existing boundary/topology/probe scripts instead of adding a parallel governance system.

## Watch Out For

- Do not execute Python or JS/TS Strategy code in web/API/Go.
- Do not let Python own backend routes, persistence, job lifecycle, Match completion, scoring, public evidence, or fallback behavior.
- Do not let stronger candidate evidence regress JS/TS counted support.
- Do not treat Docker, gVisor, subprocess, WASM/WASI, or microVM notes as production certification without required proof.
- Do not allow arbitrary PyPI/package installs.
- Do not accept stopped runtime service, stopped Python runtime, skipped container, skipped runsc, or stale artifacts as passing evidence.
- Do not expose private Strategy data in public MatchSet or replay proof outputs.
