# v1.22 Research Summary

## Findings

- v1.21's Zig false negative was caused by an empty environment preflight. `env -i zig version` fails locally, while `env -i PATH="$PATH" zig version` works.
- Local proof tooling is available: `zig 0.16.0`, `wasmtime 45.0.0`, `rustc 1.95.0`, and Rust target `wasm32-wasip1`.
- Zig standard library WASI output pulls in host capabilities that are not acceptable for the current candidate lane, including time/random/path/poll style imports. A tiny no-std Zig proof using only `fd_read` and `fd_write` keeps the WASI import boundary honest.
- Zig's target triple for the local toolchain is `wasm32-wasi`, while Rust uses `wasm32-wasip1`. Artifact validation must be language-sensitive.
- Preview 1 stdin/stdout JSON remains the least disruptive ABI for v1.22 because it is already schema-validated, stream-capped, compiler-agnostic, and aligned with runtime-service ownership.
- Direct exports need a separate memory/string allocation contract before they can replace JSON streams.
- Component model/WIT is promising long term, but toolchain maturity and cross-language parity need a dedicated milestone.

## Recommendation

Proceed with Zig as non-counted exhibition alpha only after compile, artifact metadata, Wasmtime execution, and ABI-valid behavior pass. Keep direct exports and component model/WIT as v1.23+ research candidates.
