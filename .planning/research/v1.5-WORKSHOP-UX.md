# v1.5 Workshop UX Research: Code Playground and Testing Matrix

**Project:** Coward's Game
**Milestone:** v1.5 Strategy Workshop Power Tools and Advanced Strategy Library
**Researched:** 2026-05-20
**Confidence:** HIGH for implementation direction, MEDIUM for exact UI layout pending design pass

## Findings

### 1. Treat the gauntlet as a deterministic MatchSet matrix, not a new runner

The existing Workshop already submits immutable Strategy Revisions, launches a single-opponent Workshop test, polls MatchSet status, shows scoring, and links public/owner replay views. v1.5 should generalize this into a selectable gauntlet/test matrix by reusing the current MatchSet, match job, scoring, Chronicle, replay, and privacy DTO spine.

**Borrowed pattern:** GitHub Actions matrix strategies define a compact set of axes, expand them into ordered combinations, support excludes, and expose per-combination job status. Coward's Game should use the same mental model: `candidate revisions x opponent revisions x presets/arenas/seeds/mirror side`, expanded into a canonical ordered matrix with a persisted matrix spec.

**Implementation direction:**

- Must use submitted immutable Strategy Revisions only. Draft source can validate, but tests execute only locked revisions.
- Persist a `WorkshopGauntletRun` or extended MatchSet metadata containing selected axis IDs, rule/spec/runtime versions, expanded match IDs, matrix ordering, creator, and a deterministic duplicate key.
- Generate matrix rows in stable order from explicit arrays. Never depend on insertion order from SQL without `order by`, local time, random IDs for result semantics, or client-side ordering.
- Keep execution in worker/runtime boundaries. The web/API process may validate source and create jobs, but must not invoke SoldierBrain or execute Strategy source.
- Cap matrix size with presets: Smoke, Standard, Stress. Let players select cells, but show match-count and queue cost before launch.

**UX direction:**

- Left panel: selected revision(s), baseline opponent set, preset, and toggles for mirror sides/seeds.
- Center: matrix grid. Rows are tested revisions or opponents depending on selected comparison mode; columns are opponents/presets/arenas; cells show status and result.
- Cell states should be visually distinct: queued, running, win, loss, draw, strategy failure, system failure/degraded, replay unavailable.
- Clicking a cell opens a detail drawer with match IDs, side assignment, arena, seed, outcome, failure category, scoring deltas, and replay links.
- Default first run should be small and legible: one candidate revision against the starter baseline on Smoke.

### 2. Revision comparison should combine source diff, metadata diff, and result deltas

The project already uses Monaco. Monaco supports diff editors and diagnostic markers, so v1.5 should use Monaco's diff editor for owner-only revision source comparison rather than building a custom code diff.

**Borrowed pattern:** GitHub pull request comparison combines source diff options with metadata and review context. For Coward's Game, the comparable object is not a branch; it is an immutable Strategy Revision with hash, compatibility, lineage, validation, and deterministic results.

**Implementation direction:**

- Must compare immutable revisions by ID and source hash. Never compare a mutable draft as if it were a revision.
- Must fetch source only through owner-authorized source endpoints. Public Strategy cards may link to results and hashes, but not source diff.
- Source diff must be owner-only for user Strategies. Starter and system-owned sample sources may be displayed when already public in the Workshop.
- Metadata diff should include label, notes, source hash, source bytes, valid/invalid state, validation issue counts, runtime/spec/engine compatibility, starter lineage, created timestamp, and used-in-matches count.
- Result deltas must be computed only between runs with the same gauntlet profile: same opponent set, preset, arena IDs, seed list, mirror policy, scoring policy, rule/spec/engine/runtime versions.
- If profiles differ, show "not comparable" and offer to rerun one revision on the other's profile.

**UX direction:**

- Comparison screen tabs: `Source`, `Metadata`, `Results`, `Diagnostics`.
- `Source`: split diff by default; unified diff as nice-to-have if Monaco integration is easy.
- `Metadata`: compact before/after table with changed fields highlighted.
- `Results`: delta table and matrix heatmap: points, W/L/D, strategy failures, surviving Soldiers, survival turns, per-opponent deltas.
- `Diagnostics`: compare validation/runtime failure categories and counts, not raw private memory or objective payloads.

### 3. Diagnostics should be structured, grouped, and replay-addressable

Current validation reports already include code, severity, constraint, remediation, reference, source hash, source bytes, compatibility, and forbidden patterns. v1.5 should make diagnostics more editor-native and test-run-native.

**Borrowed pattern:** Monaco markers put diagnostics inline; Playwright trace and HTML reports make failures searchable/filterable and let users drill from summary to exact step, logs, errors, source, and timeline. Coward's Game should not copy browser trace internals, but it should copy the "summary -> failing step -> replay evidence" navigation.

**Implementation direction:**

- Add optional source locations to validation issues: `startLineNumber`, `startColumn`, `endLineNumber`, `endColumn` when available from TypeScript diagnostics, regex matches, or API-shape checks.
- Use Monaco `setModelMarkers` to show validation errors and warnings inline in the editor.
- Runtime/test diagnostics should use a typed taxonomy:
  - `validation_error`
  - `transpile_error`
  - `invalid_strategy_output`
  - `invalid_action`
  - `timeout`
  - `source_or_memory_limit`
  - `strategy_failure`
  - `system_failure`
  - `chronicle_unavailable`
  - `replay_projection_unavailable`
- Include public-safe match context where available: Match ID, side, Soldier ID if already public-safe, Round, Activation, Cycle, Action type, terminal reason, result impact, and replay event sequence.
- Keep private fields out of default diagnostics: Strategy source excerpts, StrategyMemory, SoldierMemory, objective payloads, raw Awareness Grid details, stack traces, and runtime internals.
- Owner-only diagnostics may point to the user's source line and owner debug replay, but still should not serialize memory/objective payloads into public DTOs or URLs.

**UX direction:**

- Show a diagnostics summary above the matrix: first failure category, total affected matches, strategy-failure count, system-failure count, and "needs rerun" state.
- Group diagnostics by category first, then by match. A player should not have to inspect 30 duplicate timeout rows one by one.
- Each runtime diagnostic should have a primary action: `Open owner debug replay`, `Open public replay`, `Compare with previous revision`, or `Rerun matching gauntlet`.
- Use plain explanations tied to game terms: Soldier, Match, Phase, Round, Activation, Cycle, Action, Advance, STONE, FALLEN, Chronicle.

### 4. Replay handoff should use deep, privacy-preserving evidence links

The existing Workshop already exposes public replay links and owner debug replay links for Workshop participants. v1.5 should make replay handoff cell-level and diagnostic-level.

**Implementation direction:**

- Must keep public replay DTOs privacy-safe by default.
- Must keep owner debug authorization server-side. Query params may request owner debug, but the server decides whether owner data is returned.
- Add optional replay deep-link support if the replay viewer can accept it cleanly: `?eventSequence=...` or `?focus=round.activation.cycle`. This is nice-to-have for v1.5 unless the replay state model already has a stable route contract.
- Store only event sequence IDs or public event references in matrix diagnostics. Do not put private payloads in URLs.

**UX direction:**

- Matrix cell drawer: `Open replay` and, when authorized, `Open owner debug`.
- Diagnostics rows: direct link to the first relevant replay event.
- Result summary: "best win", "worst loss", and "first failure" replay links.
- Replay viewer should show whether it is public or owner debug mode exactly as v1.4/v1.1 already established.

### 5. Performance summaries should mean deterministic match performance, not host profiling

For players, "performance" should primarily mean competitive behavior across deterministic trials. Runtime timing/resource measurements are useful diagnostics, but they must not influence engine behavior or be confused with rankings.

**Implementation direction:**

- Must summarize from deterministic Match/MatchSet results and Chronicles:
  - points
  - wins/losses/draws
  - strategy failure penalties
  - surviving Soldiers
  - survival turns
  - per-opponent record
  - per-arena/seed record
  - mirror-side split
  - degraded/system-failed match counts
- Nice-to-have summaries from Chronicle events if already public-safe:
  - average Round reached
  - contraction survival rate
  - no-Advance STONE count
  - FALLEN count
  - Backstab involvement count
- Runtime budget summaries are useful but should be separate: timeout count, invalid-output count, output-size/memory-limit incidents. If wall-clock time is recorded by workers, label it as host runtime diagnostics, not game performance.
- Use stable sorting for all summaries and tie-breakers. Existing scoring already sorts by points, wins, surviving Soldiers, survival turns, then revision ID; v1.5 should reuse or extend that deterministic convention.

**UX direction:**

- Summary cards at top: record, points, reliability, survival, best/worst opponent.
- Matrix heatmap below: per-opponent/preset outcomes.
- Result delta view: current revision vs selected baseline revision.
- Avoid durable rating language. These are gauntlet results, not permanent ratings.

### 6. WebContainer-style browser execution is the wrong pattern for v1.5

WebContainers and browser-native coding sandboxes are compelling for general code playgrounds because they run Node-like tooling inside a browser tab. That is not compatible with Coward's Game's current security model and product promise.

**Decision:** Do not run Strategy code in the browser, Next.js API process, Node `vm`, or a client-side sandbox in v1.5. Use the existing worker/subprocess/container adapter boundaries. Borrow playground UX patterns, not their execution model.

## Recommended v1.5 Requirements

### Must-have

1. **Selectable Workshop gauntlet matrix**
   - User can select one or more valid immutable Strategy Revisions, a starter/seed opponent set, and a Smoke/Standard/Stress preset.
   - System expands selections into a deterministic matrix with stable ordering, explicit match count, and persisted profile metadata.
   - System creates MatchSets through existing persistence/job infrastructure and locks all participating revisions before execution.

2. **Matrix status and result grid**
   - User can see each matrix cell status and aggregate summary while jobs are queued/running/complete/degraded.
   - User can filter or group by opponent, preset, arena, seed, side, outcome, and failure category.
   - Completed cells expose replay links when available.

3. **Revision comparison**
   - User can compare two owned immutable revisions.
   - Comparison shows owner-only source diff, metadata deltas, validation deltas, and deterministic result deltas for identical gauntlet profiles.
   - If result profiles differ, UI blocks misleading deltas and offers a rerun on a matching profile.

4. **Structured diagnostics**
   - Validation reports include enough location data to place Monaco markers where possible.
   - Runtime/test failures are categorized with public-safe Match/Round/Activation/Cycle context where available.
   - Diagnostics distinguish Strategy failure from system failure.

5. **Replay handoff**
   - Matrix cells and diagnostics link to public replay and owner debug replay according to existing authorization rules.
   - Public links do not expose source, StrategyMemory, SoldierMemory, objective payloads, Awareness Grid internals, private runtime internals, or stack traces.

6. **Performance summaries**
   - User can see deterministic gauntlet summaries: points, W/L/D, failure penalties, survival metrics, per-opponent results, per-arena/seed splits, and degraded/system-failed counts.
   - Summary copy makes clear that results are tied to a specific gauntlet profile and rule/spec/runtime version.

7. **Privacy and determinism tests**
   - Public gauntlet/result/replay DTO tests assert no private fields leak.
   - Determinism tests prove identical gauntlet profiles expand to identical matrix order, match IDs or duplicate keys, scoring, and result summaries.
   - Runtime tests prove web/API routes create jobs but do not execute Strategy code.

### Nice-to-have

1. Unified diff toggle in addition to split Monaco diff.
2. Replay deep links to exact event sequence or Cycle.
3. Exportable owner-only gauntlet JSON summary for local analysis.
4. Saved named gauntlet profiles, such as `Starter Baseline`, `Mobility Stress`, and `Advanced Library Candidate`.
5. Inline remediation quick fixes for simple validation errors, only where deterministic and obvious.
6. Chronicle-derived tactical summaries, such as Backstab involvement or contraction survival, if already public-safe and cheap to compute.
7. Small multiples or sparklines for revision trend history, provided they compare only matching profiles.

## Pitfalls/Risks

### Critical

1. **Combinatorial matrix explosion**
   - Risk: selected revisions x opponents x arenas x seeds x mirrors creates hundreds of Matches and overwhelms workers/UI.
   - Mitigation: hard cell caps, preset match-count preview, Smoke default, queued status visibility, and disabled launch when over cap.

2. **Misleading result deltas**
   - Risk: comparing Revision A on Smoke against Revision B on Stress produces fake conclusions.
   - Mitigation: require identical gauntlet profile hashes for deltas; otherwise show profile mismatch.

3. **Executing Strategy code in the wrong process**
   - Risk: adding a playground-style "run in browser/API" shortcut violates core architecture.
   - Mitigation: route tests only through MatchSet jobs and StrategyExecutionAdapter workers/subprocesses/containers.

4. **Public data leaks through diagnostics**
   - Risk: failure messages, source excerpts, stack traces, memory, objectives, or owner debug context leak into public matrix/result/replay DTOs.
   - Mitigation: schema-level public DTO leak assertions, separate owner DTOs, and no private payloads in URLs.

5. **System failure scored as Strategy failure**
   - Risk: worker, persistence, or Chronicle failures unfairly penalize a Strategy.
   - Mitigation: typed failure taxonomy and score policy that keeps system failures degraded/non-counted for competitive summaries.

### Moderate

1. **Revision immutability confusion**
   - Risk: user edits draft after selecting a revision and thinks the matrix tested the draft.
   - Mitigation: prominent "tests run submitted revision hash" label and submit-before-run flow.

2. **Matrix UI becomes unreadable**
   - Risk: dense result grids hide the important result.
   - Mitigation: summary-first layout, filters, cell drawer, and caps. Avoid nested cards and long prose.

3. **Diagnostics without source locations feel disconnected**
   - Risk: validation guidance remains textual and players miss the relevant code.
   - Mitigation: add line/column locations where possible; fall back to grouped guidance when exact ranges are not available.

4. **Replay deep links become unstable**
   - Risk: event sequence query params break across Chronicle compatibility versions.
   - Mitigation: treat deep links as nice-to-have unless backed by replay compatibility tests.

5. **Host runtime timing undermines deterministic framing**
   - Risk: users overinterpret wall-clock timing as game performance.
   - Mitigation: separate "runtime budget diagnostics" from "match performance"; never feed host timing into engine/scoring.

### Minor

1. **Local time formatting noise**
   - Risk: timestamps differ by locale/timezone in tests/screenshots.
   - Mitigation: prefer ISO or server-formatted timestamps in comparison tables.

2. **Unstable labels from renamed revisions**
   - Risk: labels/notes change perception of immutable source.
   - Mitigation: compare by revision ID/source hash; treat labels as metadata only.

3. **Overbuilding saved profiles too early**
   - Risk: named profile CRUD distracts from core matrix/replay workflow.
   - Mitigation: ship fixed presets first; add saved profiles only after gauntlet behavior is proven.

## Acceptance Criteria Ideas

1. Given a valid submitted revision, when the user launches a Smoke gauntlet against selected starters, then the server creates one persisted MatchSet whose matrix expansion exactly matches the selected opponent/preset/seed/mirror axes.
2. Given the same gauntlet profile is launched twice for the same revision set, then the profile hash, expanded matrix order, match count, and scoring projection are deterministic.
3. Given a gauntlet is running, then the Workshop matrix shows queued/running/complete/degraded status without requiring a page reload.
4. Given a matrix cell has a completed Match with a Chronicle, then the cell drawer exposes `Open replay`; if the current user owns a participating Workshop revision, it also exposes `Open owner debug`.
5. Given a public replay or public gauntlet result response, then automated leak tests fail if the DTO contains Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw Awareness Grid details, owner debug, private runtime internals, or stack traces.
6. Given two owned revisions, then the comparison screen renders a Monaco source diff and a metadata delta table without exposing either source to non-owners.
7. Given two revisions have results from different gauntlet profiles, then the result delta panel refuses to compare them and identifies the mismatched axes.
8. Given validation returns line/column information, then the Workshop editor shows Monaco error/warning markers at the matching ranges.
9. Given runtime produces invalid Strategy output, timeout, or invalid Action diagnostics, then the matrix summary groups those failures by category and distinguishes them from system failures.
10. Given a degraded MatchSet caused by a system failure, then performance summaries show degraded/system-failed counts without applying Strategy failure penalties.
11. Given a revision was edited after submission, then launching a gauntlet still uses the selected immutable revision ID/source hash, not the current draft.
12. Given a cell cap is exceeded, then launch is blocked before any MatchSet or Match jobs are created.
13. Browser verification covers selecting a gauntlet, launching it, polling status, opening a matrix cell, opening replay/owner debug links, comparing revisions, viewing diagnostics markers, and verifying responsive layout.

## Agreement Points Needed From Strategy Library Research

1. **Canonical baseline opponent set**
   - Which v1.4 starter Strategies are the default v1.5 gauntlet baseline?
   - Should the baseline include all 10 starters, a smaller representative subset, or tiered profiles?

2. **Advanced Strategy candidate profile**
   - What gauntlet profile proves an Advanced Strategy is credible enough for the library?
   - Minimum acceptable evidence: Smoke only, Standard across starters, advanced-vs-advanced round robin, or all of the above?

3. **Strategy taxonomy**
   - Agree on tags/doctrine categories used in both Strategy Library cards and matrix filters: mobility, outlast, contact pressure, Backstab bait, contraction survival, memory-driven, anti-reckless, etc.

4. **Quality thresholds**
   - Define non-negotiable inclusion criteria: validates under current compatibility, no runtime failures in baseline gauntlet, distinct doctrine from existing library, readable source/comments, and replay evidence.
   - Decide whether 8-9 advanced Strategies is acceptable only when diversity and result evidence are stronger than the count target.

5. **Result-summary vocabulary**
   - Align labels used on Advanced Strategy cards and Workshop summaries: "gauntlet record", "baseline result", "advanced-only result", "reliability", and "survival". Avoid durable "rating" or "ranked" language.

6. **Example MatchSet/tournament reuse**
   - Decide whether v1.5 Workshop gauntlet runs should produce the same evidence used for Advanced Library publication and the representative 8+ entrant tournament, or whether library validation and demo tournament are separate artifacts.

7. **Privacy contract for library examples**
   - Confirm which advanced Strategy sources are intentionally public/forkable and which remain account-owned/private. Source diff and public Strategy card behavior depend on this distinction.
