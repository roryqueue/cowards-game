# v1.24 Research Summary

**Project:** Coward's Game
**Milestone:** v1.24 Runtime Abuse Lab and ABI Future-Proofing
**Domain:** Runtime abuse probes, sandbox-readiness evidence, no-fallback behavior, WASM/WASI ABI future paths, direct exports, Component Model/WIT, signed-in multi-runtime regression proof
**Researched:** 2026-05-30
**Confidence:** High for keeping WASI Preview 1 stdin/stdout JSON as the active execution ABI; high for abuse-lab and readiness-matrix requirements; medium for direct-export proof feasibility; medium for Component Model/WIT proof feasibility across Rust and Zig; low for any production sandbox certification claim without a dedicated deployment and operations proof.

## Executive Summary

v1.24 should be an evidence and claim-calibration milestone, not a product-promotion milestone. v1.23 already promoted Rust and Zig to non-counted exhibition beta, kept Python as non-counted exhibition beta, and preserved JS/TS as the only counted Strategy path. The next useful step is to make runtime abuse evidence more systematic across current lanes and to write a public-safe production-sandbox readiness matrix that states exactly what each lane proves, does not prove, and needs next.

The active WASM/WASI execution ABI should remain WASI Preview 1 stdin/stdout JSON. Rust still supports `wasm32-wasip1` as the Preview 1 target, Zig still documents `wasm32-wasi`, and Wasmtime remains a suitable runtime for WebAssembly, WASI, and Component Model experiments. WASI 0.2 and WIT are real future paths, but they should be treated as migration candidates. Direct exports are also worth probing, but they require explicit memory ownership, buffer passing, caps, schema validation, failure semantics, and language parity proof before they can replace the current envelope.

## Baseline Findings

- v1.23 is the regression floor: JS/TS counted; Python, Rust, and Zig non-counted exhibition beta; Rust/Zig backed by immutable WASM/WASI artifacts; Preview 1 stdin/stdout JSON active; direct exports and Component Model/WIT unpromoted.
- v1.23 signed-in proof completed JS/TS-vs-Rust, Rust-vs-Rust, Rust-vs-Zig, and Zig-vs-Zig exhibitions with result/replay evidence and public privacy checks.
- Existing hardening evidence covers important WASM/WASI abuse cases, but v1.24 should make the abuse taxonomy cross-runtime, compare lanes honestly, and prove no silent fallback when implementations are unavailable, stale, mismatched, malformed, or capability-invalid.
- Production sandbox certification is a stronger claim than local hostile-probe evidence. v1.24 should produce readiness criteria and gap evidence, not certification by implication.

## Stack Findings

- Rust `wasm32-wasip1` remains the correct Rust target for WASI Preview 1 module execution and replaced the older `wasm32-wasi` naming after Preview 2 work began.
- Zig documents building WASI modules with `-target wasm32-wasi`; v1.23's safe no-std path stayed within `fd_read`/`fd_write` imports.
- Wasmtime documents support for WebAssembly, WASI, and the Component Model, with configurable resource controls. That makes it a practical host for direct-export and Component Model/WIT proof spikes without changing production Match execution.
- WASI 0.2 APIs are defined with WIT and the Component Model, while WASI 0.1 remains more widely supported among runtimes. This supports keeping Preview 1 active while investigating WIT.

## Feature Findings

- The abuse lab should group probes by behavior rather than language: capability attempts, process/system access, import/package drift, filesystem/preopen access, network access, time/random access, oversized output, oversized memory, timeout/fuel, crash/trap/panic, malformed ABI payloads, invalid schemas/actions, stale/missing/mismatched artifacts, stopped/unavailable runtime lanes, and private-data redaction.
- Evidence should distinguish Strategy failure from system failure and should make unsupported runtime lanes fail loud without substituting another language, source execution, stale artifact execution, or alternate ABI execution.
- Public-safe readiness output should summarize categories and outcomes while excluding Strategy source, StrategyMemory, SoldierMemory, objective payloads, raw diagnostics, host paths, env values, tokens, DB details, package paths, and private runtime internals.
- Signed-in regression proof should explicitly include JS/TS counted behavior and non-counted exhibition beta behavior for Python, Rust, and Zig, plus result/replay page review.

## Architecture Findings

- Go should remain the owner of orchestration, persistence-facing backend behavior, Match lifecycle, scoring, public evidence, retry policy, and promotion decisions.
- Runtime-service should remain the owner of hostile Strategy execution through registered runtime implementations and schema-validated ABI envelopes.
- Direct-export proof should stay inside runtime-service/Wasmtime proof code and record memory allocation/free, input buffer ownership, output buffer ownership, caps, schema validation, trap/error mapping, and Rust/Zig parity.
- Component Model/WIT proof should define a minimal Strategy world and record WIT shape, generated bindings/tooling, Wasmtime host integration, caps, validation, guest import surface, Rust/Zig parity, and rollback impact.
- ABI decisions should be explicit and fail closed: metadata declaring a non-active ABI must not become eligible for Match execution until a future promotion decision changes the active lane.

## Watch Out For

- Do not let readiness evidence imply production sandbox certification.
- Do not promote Python, Rust, Zig, WASM/WASI, direct exports, or Component Model/WIT into counted, ranked, ladder, or gauntlet play.
- Do not execute Strategy code in web/API/Go.
- Do not let abuse probe fixtures leak private runtime internals into public DTOs or pages.
- Do not hide runtime unavailability by falling back to JS/TS, source execution, stale artifacts, or a different ABI.
- Do not treat Component Model/WIT's typed contracts as a substitute for resource caps, schema validation, privacy, replay proof, rollback, and operational evidence.
