# v1.28 Research Summary

**Project:** Coward's Game
**Milestone:** v1.28 Match Execution Operations, Recovery, and Incident Drills
**Domain:** Internal execution recovery, dead-letter/quarantine lifecycle, operator requeue/rerun controls, local incident drills, stale lease and duplicate worker recovery, private operator evidence, and compatibility with `match-execution-app-v1`
**Researched:** 2026-05-30
**Confidence:** High for scope, boundaries, and phase structure; medium for exact storage shape until Phase 201 inventories the current database and Go internal endpoint seams.

## Executive Summary

v1.28 should be an internal operations layer around the v1.26 Go execution spine. The current architecture already has the right joints: Go claims jobs, builds runtime-service requests, classifies runtime failures, records attempts, completes Matches, refreshes MatchSets, and projects public-safe evidence. This milestone should add private quarantine/dead-letter state, deterministic operator recovery controls, repeatable incident drills, and redacted operator evidence around those joints.

The milestone should not widen `match-execution-app-v1`, add public admin/result/replay UX, promote runtimes, certify a production sandbox, change counted language eligibility, migrate the WASM/WASI ABI, or assume v1.27 work from the parallel checkout.

## Baseline Findings

- v1.26 kept `match-execution-app-v1` frozen while adding Go-owned retry/failure classification and public-safe execution metadata projection.
- Go lifecycle code already has claim, lease, heartbeat, retry exhaustion, terminal failure, completion, Chronicle persistence handoff, and MatchSet refresh primitives.
- Current job state is centered on `match_jobs`, `match_job_attempts`, `matches`, `chronicles`, and MatchSet refresh; recovery should extend this source of truth, not replace it with a new queue system.
- Runtime-service should remain responsible only for hostile Strategy execution and schema-valid success/failure envelopes.
- TypeScript lifecycle persistence remains rollback/test/parity only and must not become normal recovery ownership.
- Public failure categories should stay within frozen compatibility vocabulary: `runtime_unavailable`, `timeout`, `system_failure`, `malformed_runtime_result`, `stale_artifact`, and existing public result/replay states.

## Recommended Stack Shape

- Add a small private Postgres recovery/audit layer for quarantine/dead-letter state and operator actions.
- Add Go-owned internal operations/recovery endpoints or commands beside existing internal run-once orchestration.
- Gate internal recovery controls with existing internal-token patterns such as `COWARDS_GO_BACKEND_INTERNAL_TOKEN`.
- Add a v1.28 `tsx` drill/evidence script modeled on the v1.26 reliability proof.
- Add Go integration/unit tests around recovery idempotency, stale leases, duplicate workers, public-safe detail storage, and MatchSet refresh.
- Extend boundary monitors with v1.28 proof checks for contract compatibility, ownership, privacy, and non-promotion claims.

## Top Uncertainties

- Whether quarantine is best represented as new private tables, internal metadata records, or a minimal status extension with strict public projection guards.
- Whether requeue and rerun should operate only on the original Match job or also create explicit replacement Match/MatchSet records for selected cases.
- How much live drill orchestration can be kept in deterministic local scripts before a small Go-owned internal drill endpoint is justified.
- Which runtime-service failure drills require fake local HTTP endpoints versus production runtime-service tests.
- Whether any backward-compatible public DTO addition is truly necessary. The default answer remains no.

## Recommended Phase Structure

1. Phase 201: v1.26 recovery baseline and operations gap inventory.
2. Phase 202: Dead-letter/quarantine model for exhausted and non-retryable execution jobs.
3. Phase 203: Deterministic internal requeue/rerun controls and idempotency guards.
4. Phase 204: Live failure-drill harness for Go/runtime-service/Postgres/browser topology.
5. Phase 205: Stale lease, duplicate worker, interrupted execution, and recovery drills.
6. Phase 206: Operator-only evidence, redaction, and public/private separation hardening.
7. Phase 207: Contract compatibility proof against `match-execution-app-v1`.
8. Phase 208: End-to-end signed-in operations recovery proof.
9. Phase 209: Audit, archive, commit, and tag.

## Watch Outs

- Do not let quarantine become a new public lifecycle state by accident.
- Do not allow requeue/rerun to duplicate Match completion, Chronicle persistence, scoring, or public evidence.
- Do not retry stale artifacts by recompiling or falling back to mutable source during Match execution.
- Do not expose operator evidence through public pages, public DTOs, proof pages, or fixture adapters.
- Do not give runtime-service job lifecycle, scoring, public evidence, or recovery-policy authority.
- Do not use v1.27 as hidden baseline context.
