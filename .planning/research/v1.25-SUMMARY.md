# v1.25 Research Summary

**Project:** Coward's Game
**Milestone:** v1.25 Match Execution Interface Freeze and Parallel App/Execution Contract
**Domain:** Match execution lifecycle contracts, app-facing DTO schemas, public/private evidence surfaces, fixture-backed app adapters, contract tests, drift monitors, result/replay page boundaries
**Researched:** 2026-05-30
**Confidence:** High for app-facing contract freeze scope; high for fixture and monitor value; medium for exact DTO shape until Phase 174 inventory is complete; low for any runtime promotion, sandbox certification, or execution ABI migration claim because those are intentionally out of scope.

## Executive Summary

v1.25 should freeze the app-facing Match execution interface rather than change the execution path. v1.24 left the project with strong runtime abuse evidence and calibrated non-promotion decisions, but the app still has several contract surfaces that could drift independently: Go public MatchSet summary outputs, replay metadata/evidence outputs, service schemas, fixture JSON, result/replay pages, lifecycle copy, runtime-service failure translation, and boundary monitors.

The next useful milestone is a contract freeze that lets app/result/replay UX and execution internals move in parallel. That requires canonical lifecycle semantics, versioned DTO schemas, a fixture/golden catalog, contract tests, fixture-backed app/test adapters, browser proof, signed-in proof, and a final freeze decision. It must not promote non-JS runtimes, certify a production sandbox, or migrate the WASM/WASI execution ABI.

## Baseline Findings

- v1.24 is the floor: JS/TS counted; Python, Rust, and Zig non-counted exhibition beta; Preview 1 stdin/stdout JSON active; direct exports and Component Model/WIT unpromoted; no production sandbox certification.
- `packages/spec/src/schemas.ts` currently defines service DTO schemas for public MatchSet summary, replay metadata, and replay evidence, but these are still named around service API v1.8 rather than a dedicated app-facing Match execution interface version.
- `apps/go-backend/live_backend.go` and fixture server routes already expose public MatchSet summary, replay metadata, and replay evidence, and sanitize replay payloads, but richer lifecycle/failure evidence remains spread across Go orchestration, runtime-service failure translation, and app copy.
- `apps/web/app/matchsets/[matchSetId]/page.tsx`, `apps/web/app/matches/server.ts`, and replay page/client code consume public service adapters and replay DTOs, with existing privacy and realism checks that can become fixture-backed contract gates.
- `apps/web/app/matches/replay-fixture.ts` already proves the value of fixture-backed replay rendering under test/local gates. v1.25 should generalize that pattern to MatchSet result and failure-state fixtures without creating production fallback.
- `scripts/check-boundary-monitors.ts`, `scripts/check-service-boundary-imports.ts`, and Go fixture validation already provide monitor patterns that can be extended for lifecycle vocabulary, DTO version, fixture coverage, and public/private evidence drift.

## Feature Findings

- Lifecycle contract should define both Match-level and MatchSet-level states, including queued, accepted, running, complete, failed, degraded, unavailable, retryable, and non-retryable semantics.
- Failure evidence needs categories for strategy failure, system failure, timeout, unavailable runtime, malformed runtime result, stale artifact, blocked, missing Chronicle, invalid result, and no result.
- DTO schemas should distinguish app-facing public contract fields from execution-internal runtime-service response fields and owner/test-only replay debug payloads.
- Fixture catalog should include complete, queued, running, strategy failure, system failure, timeout, unavailable runtime, malformed runtime result, stale artifact, and public-safe replay scenarios.
- Fixture-backed UI proof should test both rendered text and replay board realism: nonblank canvas, canonical starts where applicable, visible Soldiers and terrain inside declared board bounds, and no private fields.

## Architecture Findings

- Go remains the correct owner for lifecycle, scoring, public evidence, retry policy, and promotion decisions.
- Runtime-service remains the correct owner for hostile Strategy execution only through schema-validated ABI envelopes and registered runtime implementations.
- App-facing DTOs should be layered above runtime-service responses. The app should not need raw `executionResult`, private diagnostics, source, StrategyMemory, SoldierMemory, objective payloads, or runtime internals.
- Fixture-backed adapters should use the same schemas as Go/service-backed reads and should be gated by `PLAYWRIGHT_TEST`, `NODE_ENV=test`, or explicit local development flags.
- Drift monitors should fail on new lifecycle vocabulary, schema version drift, fixture scenario gaps, public/private boundary drift, backend ownership creep, runtime-service ownership creep, and production fixture fallback.

## Watch Out For

- Do not let a fixture adapter become a silent production fallback.
- Do not treat lifecycle freeze as runtime promotion evidence.
- Do not expose runtime-service diagnostics, host paths, env values, package paths, DB details, tokens, source, StrategyMemory, SoldierMemory, or objective payloads in public DTOs or rendered pages.
- Do not move Strategy execution into Go, web, API routes, tests that masquerade as production paths, or React components.
- Do not freeze execution internals that should remain replaceable behind the app-facing contract.
- Do not broaden ladder/ranking scope while freezing Match execution interfaces.

## Recommended Phase Structure

1. Phase 174: Match/App Boundary Baseline and Contract Inventory.
2. Phase 175: Canonical Match Lifecycle State Machine.
3. Phase 176: App-Facing DTO v1 Contract.
4. Phase 177: Fixture Catalog and Golden Match States.
5. Phase 178: Contract Test Harness and Drift Monitors.
6. Phase 179: Replay and Result Page Interface Hardening.
7. Phase 180: Execution/App Parallelization Adapter.
8. Phase 181: End-to-End Proof and Boundary Freeze Decision.
9. Phase 182: Audit, Archive, Commit, and Tag.
