# v1.9 Research: Go Read-Model Ownership Candidate

**Project:** Coward's Game  
**Milestone:** v1.9 Backend and Runtime Ownership Split  
**Researched:** 2026-05-22  
**Question:** Should the first v1.9 ownership move be Go read-model expansion?  
**Recommendation strength:** **STRONG YES, bounded to read-only surfaces.**

## Recommendation

Make the first v1.9 ownership move a **Go read-model expansion**, not Go mutation, runtime isolation promotion, or non-JS counted play. The project already has the exact foundation this move needs: canonical `SERVICE_API_ROUTES`, generated `service-api-v1.8` OpenAPI, TypeScript-service-generated Go fixtures, GET-only Go route inventory, topology checks, privacy monitors, and explicit v1.8 audit guidance to move only one boundary.

The move should be deliberately narrow: add one or two read-model surfaces to Go only after each surface is already represented in `@cowards/spec`, implemented in `@cowards/service`, covered by TypeScript parity fixture generation, and served by Go from validated read fixtures or a separately specified read-only data source. Go must remain a contract consumer. `@cowards/spec` and `@cowards/service` stay authoritative for route metadata, DTO schemas, privacy classes, and TypeScript behavior.

Do **not** move Go into writes in v1.9 unless the milestone separately specifies route ownership, auth/session model, jobs, migrations, rollback, transactional semantics, and public/private DTO behavior. Current evidence says Go mutation would combine too many unsettled ownership seams: sessions and CSRF/rate limits live in the web/TypeScript path, Strategy Revisions have immutability and source privacy rules, MatchSet creation triggers orchestration/jobs, and runtime execution must stay outside web/API and outside Go unless later specified.

## Must Stay TypeScript-Owned In v1.9

- Auth/session mutation: `POST /auth/session`, `DELETE /auth/session`, sign-up/sign-in/sign-out, cookie/session handling.
- Strategy Revision creation, validation, source retrieval, starter/advanced forks, and immutable revision locking.
- MatchSet creation, exhibition submission, ladder entry/schedule/admin governance, and all writes.
- Match orchestration, job claiming, retries, scoring writes, Chronicle persistence, and migrations.
- Strategy execution and runtime adapter ownership; Strategy code must not execute in web/API or Go.
- Full replay projection and owner debug authorization until route ownership is explicitly redesigned.
- Non-JS counted play and production sandbox promotion.

## Prerequisites For A Safe Go Read-Model Move

1. **Route ownership matrix.** For every Go-owned read route, document route id, method, path, auth scope, privacy class, TypeScript service method, Go handler, data source, fallback behavior, owner identity source, error shape, and rollback flag.
2. **TypeScript service first.** A route must be implemented in `@cowards/service` and validated with its canonical Zod schema before Go serves it.
3. **Generated manifest only.** Go route inventory must be generated from `SERVICE_API_ROUTES` metadata or checked against generated output; no hand-diverged paths, auth scopes, or privacy classes.
4. **Read-only enforcement.** Go tests must fail on non-GET methods, mutation route ids, persistence writes, job claims, migration imports, runtime execution imports, and source-bearing DTOs.
5. **Explicit web routing.** Any web use of Go must be behind an explicit environment flag or remote-client selection path, with no silent fallback from Go to in-process TypeScript during required-boundary checks.
6. **Public-safe default.** Go/service/topology output must reject Strategy source, StrategyMemory, SoldierMemory, objective payloads, owner debug, raw Awareness Grid, stack traces, stderr, sessions, tokens, host paths, and private runtime internals by default.
7. **Auth model for owner reads.** Owner routes need a documented identity bridge. Existing bearer-token fixture auth is acceptable for parity smoke, not for broader owner route ownership.
8. **Rollback plan.** Every Go-routed web read needs a one-switch rollback to TypeScript service behavior, with parity tests proving identical DTOs before and after rollback.

## Parity Fixture Strategy

Use the v1.8 pattern and make it stricter:

- Generate all Go fixture payloads by invoking `@cowards/service`; never hand-author expected Go DTOs as the source of truth.
- Store canonical JSON with deterministic key ordering, checksum manifest, route manifest, and embedded Go checksum references.
- For each new read route include at least: success, not found, malformed id or validation failure if applicable, privacy denial for owner routes, wrong-owner behavior, and degraded/system-failed evidence where the domain supports it.
- Compare parsed canonical DTO equality, not string-only snapshots. Preserve ordering-sensitive arrays exactly where product behavior depends on ordering.
- Extend `pnpm go:parity`, `pnpm topology:check`, and `pnpm boundary:monitors` so every v1.9 Go route is covered by fixture generation, Go handler tests, route-manifest drift checks, and privacy scans.
- Add one web-level smoke path only after service and Go parity pass. The web smoke should prove route selection and rollback behavior, not introduce new DTO transformation logic.

## Auth And Privacy Risks

| Risk | Why It Matters | Mitigation |
| --- | --- | --- |
| Owner identity drift | Current Go owner analytics uses operator-configured bearer tokens, while the web app uses cookie/session-backed auth. | Keep new v1.9 Go surfaces public by default; if owner reads expand, define a session-to-Go identity bridge and wrong-owner oracle behavior first. |
| Source exposure | Public Strategy cards may include source hashes/bytes but never source. Account/source routes are private. | Forbid `source`, `strategySource`, and source-bearing DTOs in Go fixtures and route inventory. |
| Owner debug leakage | Replay owner debug includes Awareness Grid and Soldier inactivity explanations. | Keep full replay projection TypeScript-owned; Go can serve public metadata only. |
| Existence oracles | Owner analytics wrong-owner/missing-run behavior can reveal private run ids. | Continue returning not found for wrong owner and forbid caller-supplied owner ids. |
| Public-safe diagnostics | Topology failures can leak URLs, tokens, host paths, or stack traces. | Extend existing redaction and public DTO leak checks to all new v1.9 Go diagnostics. |
| Schema drift | Go structs can silently lag TypeScript schemas. | Generate route and fixture manifests from `@cowards/spec`/`@cowards/service`; fail stale output in monitors. |

## Topology Requirements

- Keep Go as a separate local process with explicit base URL, fixture directory, and owner-token configuration.
- `pnpm topology:check -- --require-go` must fail loudly when the required Go process is unavailable.
- Optional local smoke can remain skipped by default, but required v1.9 verification should include live Go checks for every Go-owned read route.
- Diagnostics must identify component, base URL, contract version, fixture id, endpoint, auth requirement, and readiness state without leaking secrets.
- Do not add Kubernetes, service mesh, production deployment, or cloud observability work to this milestone. Local topology and rollback clarity are enough for the ownership decision.

## Mutation Non-Goals

Go mutation is unsafe in v1.9 unless separately replanned because these are not specified yet:

- Route ownership for every write endpoint.
- Cookie/session/auth propagation, CSRF behavior, rate limits, admin checks, and wrong-owner behavior.
- Transaction boundaries for Strategy Revision immutability, MatchSet creation, ladder entry, governance, and analytics runs.
- Job claiming, worker handoff, retries, idempotency, degraded/system-failed classification, and rollback after partial writes.
- Database migration ownership and compatibility with existing TypeScript persistence code.
- Runtime execution boundary and failure taxonomy for Strategy code.
- Source/privacy handling for Strategy source, StrategyMemory, SoldierMemory, objective payloads, owner debug, and Chronicle internals.
- Operational rollback that can move writes back to TypeScript without split-brain state.

## Validation Checks For v1.9

Minimum checks before calling Go read-model expansion complete:

- `pnpm contract:check`
- `pnpm contract:lint`
- `pnpm boundary:imports`
- `pnpm go:parity:generate`
- `pnpm go:parity`
- `pnpm topology:check`
- `pnpm topology:check -- --require-go --go-url http://127.0.0.1:8087` during live local validation
- `pnpm boundary:monitors`
- Focused service tests for each new TypeScript service read method
- Focused Go handler tests for success, not found, privacy, auth, method rejection, and manifest drift
- One web smoke test for any route actually switched to remote Go read-model selection

## Suggested v1.9 Phase Shape

1. **Ownership Decision And Route Matrix** - Choose Go read-model expansion and freeze the route ownership matrix/non-goals.
2. **TypeScript Service Read Surface Completion** - Add/migrate the chosen public read surface through `@cowards/service` first; likely `getPublicPlayerPage` or strengthened `getPublicStrategyPage`.
3. **Go Read-Model Expansion** - Generate parity fixtures, route manifest entries, Go handlers, and handler tests for the chosen route.
4. **Topology And Client Routing Proof** - Add explicit web/remote-client routing and rollback checks for the selected read route.
5. **Boundary Monitor Widening** - Extend import, privacy, Go parity, topology, and route drift monitors to make the new boundary hard to bypass.

## Final Position

The first v1.9 move should be **Go read-model expansion**, with **public Strategy page/card** as the best first new route and **public Player page** as the best follow-up only after TypeScript service implementation lands. Keep owner analytics narrow, keep full replay and Workshop paths TypeScript-owned, and explicitly reject Go mutations for this milestone.

This gives v1.9 a real ownership move without turning it into a backend rewrite, sandbox promotion, or non-JS runtime milestone.
